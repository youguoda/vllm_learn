# 投机解码框架实现对比

> 衔接 05-19（投机解码原理）。今天读两框架的投机解码实现，对比集成方式，并为明天（05-21）的 ngram 实验准备 prompt。

---

## 步骤 1：vLLM 投机解码实现

源码目录 `vllm/v1/spec_decode/`，方案丰富：

| 文件 | 方案 |
|---|---|
| `ngram_proposer.py` / `ngram_proposer_gpu.py` | **ngram**（无额外模型，从 prompt 找重复片段） |
| `draft_model.py` | 独立小模型 draft |
| `eagle.py` | EAGLE / EAGLE3（特征级 draft head） |
| `medusa.py` | Medusa（多解码头） |
| `dflash.py` | DFlash |

### 配置格式

vLLM 用 `--speculative-config` JSON：

```bash
# ngram 示例(无需额外权重)
--speculative-config '{"method":"ngram","num_speculative_tokens":3,"prompt_lookup_max":4}'
# EAGLE 示例(需 draft head 权重)
--speculative-config '{"method":"eagle","model":"<eagle-head-path>","num_speculative_tokens":4}'
```

## 步骤 2：SGLang 投机解码实现

源码目录 `sglang/srt/speculative/`，内置算法（`--speculative-algorithm`）：

> **Builtins: EAGLE, EAGLE3, NEXTN, STANDALONE, NGRAM, DFLASH**

配置示例：

```bash
# EAGLE
--speculative-algorithm EAGLE --speculative-num-steps 3 \
  --speculative-eagle-topk 4 --speculative-num-draft-tokens 8
# NGRAM
--speculative-algorithm NGRAM
```

## 两框架对比表

| 对比点 | vLLM | SGLang |
|---|---|---|
| ngram | `--speculative-config '{"method":"ngram",...}'` | `--speculative-algorithm NGRAM` |
| EAGLE | `--speculative-config '{"method":"eagle",...}'` | `--speculative-algorithm EAGLE/EAGLE3` |
| 配置方式 | **JSON 字符串**（灵活但繁琐） | **命名参数**（直观） |
| 内置算法 | ngram/draft/eagle/eagle3/medusa/dflash | EAGLE/EAGLE3/NEXTN/STANDALONE/NGRAM/DFLASH |
| 集成定位 | 可选插件 | 可选插件 |
| 调度集成 | 调度器感知 draft token 预算（05-13 schedule 的 spec_token_ids） | 类似 |

> **主要差异**：集成点基本**对称**，都是"可选插件"非默认开启。配置风格不同——vLLM 用 JSON（一个参数装所有），SGLang 用一组命名参数（更易读）。两者都支持 ngram 和 EAGLE 系列。SGLang 多了 NEXTN（DeepSeek 的 Multi-Token Prediction）。

## 步骤 3：接受率 → 加速比公式

```
设: K = 每次猜测的 draft token 数
    α = 接受率(每个 draft 被接受的概率)

平均每次验证接受的 token 数(期望):
    n_accept = (1 - α^(K+1)) / (1 - α)    [几何级数, 第一个拒绝即停]
    简化(α 不太接近1): ≈ 平均接受 K×α 个左右

理论加速比 ≈ n_accept / (1 + c)
    c = 验证开销相对单步 decode 的比例(通常很小, 因访存瓶颈)

例: K=3, α=0.8:
    n_accept = (1-0.8^4)/(1-0.8) = (1-0.41)/0.2 ≈ 2.95 → 约 2.4 个/次
    加速比 ≈ 2.4x (理想, 实际受验证开销和 batching 影响)
```

> **关键**：接受率 α 是命门。α=0.8 → ~2.4x；α=0.5 → ~1.5x；α 太低（自由创作 ngram 命中率可能 <0.2）→ 几乎无加速甚至变慢（验证开销白付）。

## 步骤 4：明天实验准备

[prepare_spec_prompts.py](../prepare_spec_prompts.py) 生成两类 prompt（存 `spec_prompts.json`）：

- **高重复**（代码、模板、固定格式列表）：ngram 命中率高 → 投机受益大
- **低重复**（科幻故事、自由分析）：ngram 命中率低 → 投机受益小甚至负

### 自测：ngram 投机的 draft 来源？

> **答**：**不是小模型**——是从**当前请求的上下文（prompt + 已生成部分）里找与当前位置相似的 n-gram 片段，把其后续 token 当作 draft**。比如生成代码时 "def bubble_sort(" 这种结构在 prompt 里出现过，ngram 直接把后面的 token 拿来当猜测。
>
> 所以：**代码/模板/重复列表**（大量重复语法结构）命中率高；**自由创作**（每句都新）命中率极低。ngram 是"零成本"投机（无需额外模型/训练），但只在高重复场景有效——这正是明天实验要验证的。

## 一句话

> vLLM 用 `--speculative-config` JSON，SGLang 用 `--speculative-algorithm` 命名参数，两者都支持 **ngram(零成本) + EAGLE(高接受率)**，集成对称。ngram 的 draft 来自上下文重复片段——**代码/模板受益大，自由创作几乎无效**。加速比由接受率决定（α=0.8 → ~2.4x）。
