## 步骤 2：调度主循环

### 关键设计：没有"prefill 阶段"和"decode 阶段"之分

源码注释（scheduler.py:330）道破了 vLLM v1 调度的精髓：

> "There's no 'decoding phase' nor 'prefill phase' in the scheduler. Each request just has `num_computed_tokens` and `num_tokens_with_spec`. At each step, the scheduler tries to assign tokens so that each request's `num_computed_tokens` can catch up. This is general enough to cover **chunked prefills, prefix caching, speculative decoding**."

> **顿悟**：vLLM 把"prefill 算 prompt"和"decode 生成"统一成了一件事——**"让 num_computed_tokens 追上 num_tokens"**。prefill 是一次追很多 token，decode 是一次追 1 个。这个统一抽象让 chunked prefill / 投机解码 / 前缀缓存全部自然融入同一循环。

### token_budget：Continuous Batching 的核心闸门

```python
token_budget = self.max_num_scheduled_tokens   # 每步的 token 预算(=max_num_batched_tokens)

# 先调度 RUNNING 请求
while req_index < len(self.running) and token_budget > 0:
    num_new_tokens = request.num_tokens_with_spec - request.num_computed_tokens
    # long_prefill 截断: 长 prompt 一次最多算这么多(chunked prefill!)
    if 0 < long_prefill_token_threshold < num_new_tokens:
        num_new_tokens = long_prefill_token_threshold
    num_new_tokens = min(num_new_tokens, token_budget)   # 不超预算
    ...
    token_budget -= num_new_tokens
```

> 这就是 [[vLLM-Continuous-Batching实验-06-11]] 里"每步 token 数有上限"的代码实现。`token_budget` 在 running 和 waiting 之间共享——**保证长 prefill 不会把整个 step 占满、饿死正在 decode 的请求**。`long_prefill_token_threshold` 就是 Chunked Prefill 的开关：长 prompt 被切成多步，每步最多算 threshold 个。

### 先 running 后 waiting 的顺序

```python
# First, schedule the RUNNING requests.   (line 364)
while req_index < len(self.running) and token_budget > 0: ...
# Then, schedule the WAITING requests.    (line 548)
while (self.waiting or self.skipped_waiting) and token_budget > 0: ...
```

> **先保运行中的请求继续 decode，再用剩余预算接纳新请求**。这保证已开始的请求平稳推进（低 ITL），新请求在有余力时才入场。

## 步骤 3：抢占逻辑（preemption）

### 触发点：allocate_slots 返回 None

running 请求要新 KV 块，但 `allocate_slots` 分配不到（free 块耗尽）：

```python
while True:
    new_blocks = self.kv_cache_manager.allocate_slots(request, num_new_tokens, ...)
    if new_blocks is not None:
        break                              # 分到了, 继续
    # 分不到 → 抢占最低优先级的请求
    if self.policy == SchedulingPolicy.PRIORITY:
        preempted_req = max(self.running, key=lambda r: (r.priority, r.arrival_time))
    else:
        preempted_req = self.running.pop()  # FCFS: 抢占队尾(最晚到的)
    self._preempt_request(preempted_req, scheduled_timestamp)
    if preempted_req == request:
        break                              # 连自己都抢了还不够, 放弃
```

> FCFS 策略下抢占 `self.running.pop()`——**队尾，即最晚到达的请求**。一直抢到能腾出块为止；如果抢到自己头上还不够，就放弃调度本请求。

### _preempt_request：vLLM v1 是 RECOMPUTE-only（重要发现）

```python
def _preempt_request(self, request, timestamp):       # line 929
    self.kv_cache_manager.free(request)                # ① KV 块直接释放回 free_list
    self.encoder_cache_manager.free(request)
    request.status = RequestStatus.PREEMPTED
    request.num_computed_tokens = 0                    # ② 计算进度清零 → 下次从头 prefill!
    request.num_preemptions += 1
    self.waiting.prepend_request(request)              # ③ 插回 waiting 队首(优先恢复)
```

> **关键发现**：vLLM v1 的抢占是 **RECOMPUTE-only**——没有 SWAP-to-CPU！
> - 计划里画的"SWAPPED（KV 换到 CPU）"分支在 v1 引擎里**已被移除**。v0 有 swap，v1 简化为只重算。
> - 被抢占请求的 KV **直接 free 回 free_list**，`num_computed_tokens=0` 意味着下次轮到它要**从头重新 prefill**（代价：重算一次 prefill）。
> - 插回 waiting **队首**（prepend），所以恢复优先级高，不会饿死。
>
> 为什么 v1 砍掉 swap？因为 swap 到 CPU 再换回的 PCIe 传输，在很多场景下并不比直接重算快（尤其小模型/短前缀），且实现复杂。这和 [[KVCache设计哲学-HiCache-06-14]] 里 SGLang 用 HiCache 主动换 CPU 形成对比——**vLLM 选择"重算"的简单路线，SGLang 选择"换出 CPU"的省算路线**。

## 步骤 4：实测触发抢占（38 次）

参数：`--max-model-len 2048 --gpu-memory-utilization 0.40 --max-num-seqs 128 --no-enable-prefix-caching`，KV 池仅 **41,888 token**（并发上限 20.45x）。用 [trigger_preempt.py](../trigger_preempt.py) 思路发 80 个唯一长 prompt（ignore_eos 强制各生成 512 token）。

实测峰值：

```
Running: 51 reqs, Waiting: 29 reqs, GPU KV cache usage: 99.0%
```

抢占证据（默认 log level 不打印 "Preempted"，但 metrics 计数器是铁证）：

```
vllm:num_preemptions_total = 38.0
```

> **38 次抢占**！KV 使用率冲到 99% 时，已 running 的请求要继续 decode 但没块了，触发抢占——被抢的请求 free 掉 KV、清零进度、插回 waiting 队首，等有块了从头重算。
>
> 一个观察：**vLLM v1 的admission control 很保守**——它更倾向于把新请求留在 waiting 队列（51 running + 29 waiting），而不是激进 over-admit 再大量抢占。抢占是"running 请求 decode 撞墙"时的最后手段，不是常态。

## 自测

**Q：抢占时 recompute 策略下，该请求的 KV blocks 去哪了？**

> **答**：直接释放回 free_list（`kv_cache_manager.free(request)`），且 `num_computed_tokens` 清零。请求插回 waiting 队首（prepend）。下次轮到它时，因为进度清零，要**从头重新 prefill**——代价是重算一次 prefill 的算力。vLLM v1 没有 swap-to-CPU 选项（v0 有，v1 简化掉了），所以抢占 = 纯重算。

## 关键发现

1. **统一抽象**：v1 调度无 prefill/decode 之分，都是"让 num_computed_tokens 追上"。
2. **token_budget 共享**：running 和 waiting 共用预算，长 prefill 不饿死 decode（Chunked Prefill）。
3. **RECOMPUTE-only**：v1 抢占只重算不换出（v0 的 swap 被砍），与 SGLang HiCache 的"换出 CPU"路线相反。
