# 投机解码原理

> 投机解码（Speculative Decoding）要解决的问题：decode 阶段 GPU 算力闲置。今天搞懂 draft-verify 范式，以及 EAGLE/Medusa 两个代表方案。

---

## 步骤 1：动机——decode 阶段 GPU 在"摸鱼"

实测单请求长 decode（Qwen2.5-1.5B，600 token）：

```
单请求 decode: 600 token / 6.7s = 89.6 tok/s
对比 batch=10 (04 实验): ~849 tok/s 总吞吐
```

> **为什么单请求才 90 tok/s，batch=10 能到 849？** 因为 decode 是**访存瓶颈**：每生成 1 个 token 都要把整个模型权重（1.5B≈3GB）从显存读一遍。单请求时，这 3GB 搬运只服务 1 个 token，GPU 的并行算力全闲着等串行依赖；batch 大时同一次搬运服务多个 token，算力才被喂饱。
>
> **诚实说明**：`nvidia-smi` 的 `utilization.gpu` 在 decode 时报 95%，但这是误导——它只统计"有 kernel 在跑的时间占比"，不是**算力饱和度**。真正反映 decode 闲置的是吞吐数字（90 tok/s 远低于 GPU 的 FLOPS 能力），不是 nvidia-smi 的百分比。SM occupancy / MFU 才是对的指标，但 nvidia-smi 不暴露。

**投机解码的动机**：decode 每步只生成 1 token，串行依赖让 GPU 并行算力浪费。能不能"一次多生成几个"？

---

## 步骤 2：draft-verify 范式

核心：用一个**小模型/简单方法**先猜 K 个 token，再用**大模型一次并行验证**。

```
传统 decode (串行, 3 步延迟):
  大模型 → token_1 → 大模型 → token_2 → 大模型 → token_3
  3 次前向, 每次等上一步

投机解码 (draft + verify):
  ① draft: 小模型猜 [t1?, t2?, t3?]   ← 小模型快, 3 步但便宜
  ② verify: 大模型一次前向验证 3 个位置
       → 每个位置输出大模型认为对的 token
       → 前 k 个猜对 → 接受; 第一个猜错后全丢弃, 从那重猜
```

> **关键洞察**：投机解码省的**不是"每步计算量"**，而是**减少串行步数**。接受 K 个 draft = 做了 K 步 decode 却只花约 1 步的时间（1 次大模型前向）。代价：验证步比正常 decode 略重（同时算 K+1 个位置）。
>
> **为什么验证 K 个 token ≈ 生成 1 个的成本？** 因为验证是把 K+1 个位置**并行**丢进 Transformer（attention 本来就并行算所有位置）。decode 是访存瓶颈——权重搬运成本固定，多算几个位置几乎不增加时间。这正好把"摸鱼的算力"利用起来了。

### 正确性保证

> 投机解码用 **推测采样（speculative sampling）** 算法保证：**最终输出分布与直接用大模型完全相同**。不是近似、不是质量妥协——数学上等价。接受/拒绝的概率精心设计，使得整体采样分布无偏。这是它能被工业界采用的前提。

## 步骤 3：EAGLE vs Medusa

### EAGLE（arXiv:2401.15077）

- **普通 draft 模型的问题**：用独立小模型（7B 配 1B draft）猜，小模型和大模型**特征空间不对齐**，接受率不高。
- **EAGLE 改进**：不训独立小模型，而是训一个"**特征级 draft head**"——输入大模型最后一层的**隐状态**（特征向量，而非 token embedding），预测下一步隐状态，再用大模型的 LM head 解码。
- **效果**：draft head 直接复用大模型特征，**接受率 ~80%**，加速比稳定（2-3x）。

### Medusa（arXiv:2401.10774）

- **不用 draft 模型**，而是在大模型顶部加**多个平行解码头**：head_1 预测 t+1，head_2 预测 t+2，head_3 预测 t+3…
- 一次前向**同时猜多步**，再用**树状验证**（多个候选组成树，一次验证整棵树）。
- 优点：无需独立 draft 模型，改动小；缺点：多头需要额外训练，接受率略低于 EAGLE。

### 一句话对比

| 方案 | 核心创新 | draft 来源 |
|---|---|---|
| 普通 SD | 小模型猜 + 大模型验 | 独立小模型 |
| **EAGLE** | 特征级 draft head（复用大模型隐状态） | 大模型特征 + 轻量 head |
| **Medusa** | 多平行解码头 + 树状验证 | 大模型顶部多头 |

## 步骤 4：时序对比图

```
普通 decode:   [大][大][大][大][大]          ← 串行, 每步独立, GPU 摸鱼
                t1  t2  t3  t4  t5

Draft+Verify:  [小小小][大-验证3][小小小][大-验证]   ← 猜3步 + 1次验证
                猜      ↑接受k个   继续猜

EAGLE:         [大-出特征]→[draft head×3]→[大-验证3+1]  ← 复用特征, draft 质量高
                            ↑轻量, 接受率~80%

Medusa:        [大 + head1 + head2 + head3]→[树状验证]   ← 一次前向同时猜多步
                ↑无独立 draft 模型
```

## 自测

**Q：为什么验证 K 个 draft token 的成本 ≈ 生成 1 个 token？**

> **答**：验证是把 K+1 个位置**并行**丢进 Transformer（attention 并行算所有位置）。decode 阶段是**访存瓶颈**——时间主要花在搬运模型权重（固定成本），计算量取决于 batch size（K+1 个 token），不是 K 次串行前向。所以一次验证 K 个 ≈ 一次正常 decode 的时间。这恰好把 decode 时闲置的并行算力利用起来——**投机解码本质是"用闲置算力换串行步数"**。
