# PagedAttention 实测：亲眼看到「显存→块→并发」这条链

> 本文在 RTX 3060 12GB (WSL2) 上，通过实测 vLLM 启动日志和 `/metrics` 端点，把 PagedAttention 论文里的 "显存 → block → 并发" 这条抽象链路落到真实数字上。再用纸面推演理解碎片为什么消失、Block Table 如何运转。
>
> 配套论文笔记见 [[论文笔记-PagedAttention-vLLM]]。

---

## 第 1 步：亲眼看到 KV cache 被「分块」

### 操作

```bash
HF_HUB_OFFLINE=1 NO_PROXY="*" vllm serve Qwen/Qwen2.5-1.5B \
  --max-model-len 4096 --gpu-memory-utilization 0.85
```

### 从日志抓到的两行

```
GPU KV cache size: 228,288 tokens
Maximum concurrency for 4,096 tokens per request: 55.73x
```

> **注意**：vLLM 0.22.1 的日志格式变了，不再直接打印 `# GPU blocks: XXXX`（那是旧版 V0 引擎的格式）。新版直接报 **token 数**。但块数可以从 `/metrics` 端点拿到铁证。

### 从 `/metrics` 拿到的铁证

```bash
NO_PROXY="*" curl --noproxy '*' -s http://127.0.0.1:8000/metrics | grep cache_config_info
```

关键字段：

```
vllm:cache_config_info{... block_size="16" ... num_gpu_blocks="14268" ...}
```

### 验算：显存 → 块 → 并发

```
                  ┌─────────────────────────┐
  可用 KV 显存     │      6.1 GiB            │   ← 日志: Available KV cache memory
                  └───────────┬─────────────┘
                              │ ÷ (每 token 的 KV 字节数)
                              ▼
                  ┌─────────────────────────┐
  GPU blocks      │   14268 块               │   ← /metrics: num_gpu_blocks
                  └───────────┬─────────────┘
                              │ × block_size(16)
                              ▼
                  ┌─────────────────────────┐
  总 token 容量    │   228,288 tokens        │   ← 14268 × 16，正好 = 日志值 ✅
                  └───────────┬─────────────┘
                              │ ÷ max_model_len(4096)
                              ▼
                  ┌─────────────────────────┐
  理论最大并发     │   55.73x                │   ← 228288 / 4096，正好 = 日志值 ✅
                  └─────────────────────────┘
```

| 环节 | 数值 | 来源 |
|---|---|---|
| 可用 KV cache 显存 | 6.1 GiB | 启动日志 |
| **GPU blocks** | **14268 块** | `/metrics` 实测 |
| block_size | 16 tokens | `/metrics` 实测 |
| 总 token 容量 | 14268 × 16 = **228,288** | 验算 = 日志值 ✅ |
| 理论最大并发 | 228288 ÷ 4096 = **55.73x** | 验算 = 日志值 ✅ |

**这条链完全闭合。** "显存决定块数，块数决定能存多少 token，token 数除以单请求长度决定并发"——不再是论文里的抽象描述，而是手里这张卡上的真实数字。

## 第 2 步：用实验感受「按需分配」

跨 4 种配置启动 server，抓 `num_gpu_blocks` 和并发数。用脚本 [05_kvcache_blocks_scan.py](../05_kvcache_blocks_scan.py) 自动化。

### 实测数据表

| max_model_len | gpu_util | GPU blocks | block_size | KV tokens | blocks×16 | 最大并发 | tokens÷len |
|---:|---:|---:|---:|---:|---:|---:|---:|
| 2048 | 0.85 | **14268** | 16 | 228,288 | 228,288 | 111.47x | 111.47x |
| 4096 | 0.85 | **14268** | 16 | 228,288 | 228,288 | 55.73x | 55.73x |
| 8192 | 0.85 | **14268** | 16 | 228,288 | 228,288 | 27.87x | 27.87x |
| 4096 | 0.70 | **10055** | 16 | 160,880 | 160,880 | 39.28x | 39.28x |

### 三个观察（全部验证成功）

**观察 1：固定 gpu_util，改 max_model_len → 块数不变，变的是「每个请求占多少块」**

```
max_len=2048:  块池 14268 块 ───┐
max_len=4096:  块池 14268 块 ───┼─→ 块总数恒定！由"剩余显存"决定，与 max_len 无关
max_len=8192:  块池 14268 块 ───┘

但并发数严格反比:
  2048 → 111.47x   (每请求最多占 2048/16 = 128 块)
  4096 →  55.73x   (每请求最多占 4096/16 = 256 块)
  8192 →  27.87x   (每请求最多占 8192/16 = 512 块)
  ↑ max_len 翻倍，并发减半
```

这就是「**按需分配**」的本质：vLLM 不会因为你设了 `max_len=8192` 就预先吃掉更多显存。块池大小始终由剩余显存决定；`max_len` 只是规定了「单个请求**最多**能切走多少块」。

**观察 2：固定 max_len，调低 gpu_util → 块池缩水**

```
gpu_util=0.85:  14268 块  (KV 显存预算 6.1 GiB)
gpu_util=0.70:  10055 块  (KV 显存预算更小)
                 ↓
                块数下降 ≈ 30%，并发也随之 55.73x → 39.28x
```

`gpu_util` 是「总闸门」——它先划走一部分显存装模型权重，剩下的全部变成块池。

**观察 3：`blocks × block_size == 日志 KV tokens` 在 4 种配置下恒成立**

无一例外，验证了 token 容量就是块数的直接换算。

## 第 3 步：看碎片为什么消失（纸面推演）

设想一个真实场景：**发 10 个请求，每个实际只生成 50 个 token**（短问答很常见）。模型 `max_len=4096`。

### 传统方案：预留 max_len 连续显存

```
请求 1: [■■■■■░░░░░░░░░░░░ ... ░░░] ← 预留 4096，实际用 50，浪费 4046
请求 2: [■■■■■░░░░░░░░░░░░ ... ░░░]
  ...                              每个请求都要一整段 4096 的连续显存
请求 10:[■■■■■░░░░░░░░░░░░ ... ░░░]

预留总量 = 4096 × 10 = 40,960 tokens
实际使用 =   50 × 10 =    500 tokens
浪费     = 40,460 tokens  →  98.8% 被白白占着！
```

问题不止"浪费"。更糟的是**外部碎片**：因为要求"连续"，即使总空闲显存够，也可能找不到一段连续的 4096 空间来塞新请求。

### PagedAttention：按需分块

```
block_size = 16，每请求实际 50 token → 需要 ⌈50/16⌉ = 4 块（最后一块只用了 2 格）

请求 1: [blk_a][blk_b][blk_c][blk_d·]   ← 4 块 = 64 token 空间，浪费 14
请求 2: [blk_e][blk_f][blk_g][blk_h·]      这些块可以散落在显存任意位置
  ...                                      （非连续！靠 block table 间接寻址）
请求 10:[blk_..][.. ][.. ][.. ·]

分配总量 = 64 × 10 = 640 tokens
实际使用 = 500 tokens
浪费     = 140 tokens → 21.9%，且每请求最多浪费 block_size-1 = 15 token
```

### 两张内存条并排对比（这就是「架构图」）

```
传统方案（连续预留，外部碎片严重）:
┌────────────────────────────────────────────────────────────┐
│ R1: ■■░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░ (4096)  │
│ R2: ■■░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░ (4096)  │
│  ...                                                         │
│ 显存被预留撑满，第 56 个请求就进不来了（228288/4096≈55）       │
└────────────────────────────────────────────────────────────┘
   ■=实际用  ░=预留但闲置（98.8% 浪费）

PagedAttention（非连续小块，只剩内部碎片）:
┌────────────────────────────────────────────────────────────┐
│ ■■■■· ·■■■■ ■■■■· ■■■■■ ·■■■■ ■■■■· ■■■■■ ... 紧凑排列        │
│ 每块 16 token，按需取用，最后一块的零头是唯一浪费              │
│ 同样显存能塞下 3567 个这种请求！                              │
└────────────────────────────────────────────────────────────┘
   ■=实际用  ·=最后一块的零头（≤15 token/请求）
```

### 容量对比（最震撼的数字）

**同样 228,288 token 的 KV cache，能放多少个「50-token 请求」？**

| 方案 | 每请求占用 | 能放下 | 倍数 |
|---|---:|---:|---:|
| 传统（预留 max_len） | 4096 token | **55 个** | 1x |
| PagedAttention（按需分块） | 64 token | **3567 个** | **65x** |

> 碎片不是"优化掉一点点"，而是**把 98.8% 的浪费压到 21.9%，等效并发容量翻 65 倍**。这就是 vLLM 吞吐碾压传统框架的根本原因——不是 kernel 更快，而是**同样显存能同时服务的请求多了一两个数量级**。

## 第 4 步：Block Table 完整流程图

用一个具体句子手推一遍：**用户输入 "今天天气如何？"（假设 tokenize 成 6 个 token），模型回答到 EOS**。`block_size=16`。

### 阶段 0：请求到达，prefill

prompt = 6 个 token，⌈6/16⌉ = 1 块就够。KV Cache Manager 从 free list 取 1 个物理块（假设拿到 #102）。

```
Block Table (请求 R1):
┌────────────┬──────────────┬─────────────────┐
│ logical id │ physical id  │ filled / 16     │
├────────────┼──────────────┼─────────────────┤
│     0      │    #102      │   6 / 16        │  ← prompt 6 token 写入
└────────────┴──────────────┴─────────────────┘

物理块 #102: [今][天][天][气][如][何][·][·][·][·][·][·][·][·][·][·]
                                    ↑ 还剩 10 格空位
free list: [#103, #104, #105, ...] （#102 已被取走）
```

### 阶段 1：decode 第 1 个输出 token "今"（第 7 个 token）

最后一个 logical block (#102) 还有空位（6<16），**直接写进去，不分配新块**。

```
物理块 #102: [今][天][天][气][如][何][今][·]...  filled = 7/16
```

### 阶段 2：decode 持续，直到第 16 个 token 写满 #102

```
物理块 #102: [今][天][天][气][如][何][今][天][...全满...]  filled = 16/16  ← 满了！
```

### 阶段 3：decode 第 17 个 token → 触发「按需分配新块」

logical block 0 满了，KV Cache Manager 从 free list 再取 1 块（#103），block table 新增一行。

```
Block Table (请求 R1):
┌────────────┬──────────────┬─────────────────┐
│ logical id │ physical id  │ filled / 16     │
├────────────┼──────────────┼─────────────────┤
│     0      │    #102      │  16 / 16        │  ← 满
│     1      │    #103      │   1 / 16        │  ← 新分配，写入第 17 个 token
└────────────┴──────────────┴─────────────────┘

free list: [#104, #105, ...] （#103 也被取走）
```

> **关键**：物理块 #102 和 #103 在显存里**不需要相邻**。attention kernel 靠 block table 做「**间接寻址**」，按逻辑顺序去非连续的物理位置读 KV。

### 阶段 4：生成 EOS → 请求结束，块归还 free list

```
模型生成 <EOS>，请求 R1 完成。
KV Cache Manager 把 R1 占用的物理块全部归还:

  #102 → free list
  #103 → free list

free list: [#102, #103, #104, #105, ...] ← 立即可被下一个请求复用！
Block Table (R1): 销毁
```

这一步就是 [[vLLM-Continuous-Batching实验-06-11]] 里说的"已完成序列立即移除、释放 KV cache" —— **块归还是即时的，腾出的空间马上能给 waiting queue 里的新请求**。

### 论文 4 个要点在这个流程里的体现

| 要点 | 在流程哪一步体现 |
|---|---|
| **① 间接寻址** | 阶段 3：#102/#103 非连续，attention 靠 block table 找 KV |
| **② 按需分配** | 阶段 0/3：prompt 只取 1 块；写满才取第 2 块，绝不预留 max_len |
| **③ Copy-on-Write (CoW)** | 本例单序列没触发；若 parallel sampling 共享 prompt 块，写入时才复制（见论文笔记 §4） |
| **④ 空闲池 (free list)** | 阶段 0/4：分配从 free list 取，EOS 后归还 free list，立即可复用 |

## 第 5 步：自测收尾

### 我现在能回答的问题

**Q1：vLLM 启动时报的 "Maximum concurrency 55.73x" 是怎么来的？**
> KV cache 总容量 228,288 token ÷ max_model_len 4096 = 55.73。即"如果每个请求都用满 4096 上下文，最多能同时跑 55.7 个"。

**Q2：我把 max_model_len 从 4096 改成 8192，显存占用会翻倍吗？**
> 不会。块池大小（14268 块）由剩余显存决定，与 max_len 无关。改大 max_len 只是让单个请求**最多**能占更多块，于是并发上限从 55.73x 降到 27.87x。显存占用不变。

**Q3：block_size 为什么是 16，不是 1 或 1024？**
> 太小（如 1）→ 块数爆炸、attention 读取并行度低、block table 开销大；太大（如 1024）→ 内部碎片严重（短请求也要占一大块）、共享概率低。论文实验得出 16 是甜点（详见论文笔记 §3.3）。

**Q4：为什么 PagedAttention 能消除外部碎片？**
> 因为所有物理块**大小相同（16 token）**，分配时不需要找"连续的大段空间"，任何一个空闲块都能用。外部碎片（总量够但不连续）从根上消失，只剩每个请求最后一块的内部碎片（≤15 token）。

**Q5：一个请求生成完 EOS 后，它的显存什么时候能给别人用？**
> 立即。EOS 触发后 KV Cache Manager 把该请求所有物理块归还 free list，下一个 scheduler step 就能分给 waiting queue 的新请求。这是 Continuous Batching 高吞吐的基础。

### 核心收获

```
显存预算 (gpu_util)
    │ 划走模型权重后，剩余全部变成 ↓
块池 (num_gpu_blocks，固定)
    │ × block_size(16) ↓
token 总容量 (固定)
    │ ÷ max_model_len ↓
并发上限 (随 max_len 反比变化)
```

**一句话总结**：PagedAttention 把"为每个请求预留连续大段显存"改成"按需取用固定小块 + block table 间接寻址"，用 ≤15 token 的内部碎片换掉了 98.8% 的预留浪费，让同样的显存能服务多一两个数量级的请求。
