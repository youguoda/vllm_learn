## 异常点 / 关键拐点解释

### p1 吞吐：并发 16→32 增速放缓
> 吞吐增长比从 1→4 的 3.8x 降到 16→32 的 1.3x。**GPU 接近饱和**——并发 8-16 后新请求开始排队，加并发只增 batch 不增吞吐。vLLM 1443 vs SGLang 1342 @并发32（vLLM 略优 7%）。

### p2 TTFT：并发 32 时 vLLM 跳升
> 并发 1-16 TTFT 平稳（30-43ms），并发 32 时 vLLM 跳到 67-107ms（多轮测量有波动）。**调度器开始排队**：KV blocks 不够同时容纳所有请求的 prefill，部分请求等待。

### p3 TTFT vs 输入：完美线性
> 两框架 TTFT 随输入长度近似线性（vLLM 37→212ms，SGLang 33→206ms，输入 64→1500）。**prefill 计算量 ∝ 输入 token 数**，这是 prefill 计算密集的直接体现。斜率一致 = prefill 算法无本质区别。

### p4 多轮对话：两框架都平稳（关键认知）
> **两框架 TTFT 都不随轮数增长**（vLLM ~44-51ms，SGLang ~31-40ms @并发8）。原因：**两框架都有前缀缓存**（vLLM APC + SGLang RadixCache），第 2 轮起命中 system prompt + 历史。SGLang 全程略低（约 10-25%），但非数量级差异——因为这是"对话内复用"，不是"跨请求共享同一前缀"（后者才是 SGLang 2.1x 的场景，见 05-10）。

## 数据质量说明（诚实记录）

1. **latency 数据修正**：05-28 初版因 filler 文本重复触发 prefix cache，TTFT 平坦失真；05-30 改用唯一随机输入后呈现正确线性。
2. **3 轮一致性极好**：吞吐 3 轮 1440/1443/1444，证明测量稳定可信。
3. **单卡限制**：32K 长上下文、多卡并行、EAGLE 未实测，引用社区数据（已在对应笔记标注）。
