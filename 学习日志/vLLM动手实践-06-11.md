## 第二步：基础离线推理

### 目标

用 `vllm.LLM` 加载 Qwen2.5-1.5B 模型，跑一次批量推理。

### 最终代码 (`01_basic_inference.py`)

```python
"""
vLLM 基础离线推理 — 验证安装正确性
模型: Qwen/Qwen2.5-1.5B (约 3GB, RTX 3060 12GB 轻松运行)

注意: WSL2 下 vLLM 使用 spawn 多进程, 必须用 if __name__ == '__main__' 保护
"""

from vllm import LLM, SamplingParams


def main():
    # 1. 加载模型 — vLLM 会自动从 HuggingFace 下载
    model_name = "Qwen/Qwen2.5-1.5B"
    print(f"Loading model: {model_name} ...")
    llm = LLM(
        model=model_name,
        max_model_len=4096,          # 默认 131072 太大, 12GB 卡放不下 KV cache
        gpu_memory_utilization=0.85,  # 留一些给桌面和系统
    )

    # 2. 准备 prompts
    prompts = [
        "Hello, my name is",
        "The capital of France is",
        "The future of AI is",
    ]

    # 3. 配置采样参数
    sampling_params = SamplingParams(
        temperature=0.7,
        top_p=0.9,
        max_tokens=64,
    )

    # 4. 批量推理
    print("\n--- Generating ---\n")
    outputs = llm.generate(prompts, sampling_params)

    for output in outputs:
        prompt = output.prompt
        generated_text = output.outputs[0].text
        print(f"Prompt:    {prompt!r}")
        print(f"Generated: {generated_text!r}")
        print()


if __name__ == "__main__":
    main()
```

### 踩坑记录

#### 坑 1：WSL2 多进程 spawn 错误

**错误信息：**
```
RuntimeError:
    An attempt has been made to start a new process before the
    current process has finished its bootstrapping phase.
    ...
    if __name__ == '__main__':
        freeze_support()
```

**原因：**

vLLM 内部使用多进程来运行 EngineCore。在 Linux 上默认用 `fork`，但 WSL2 的 NVML 不兼容 `fork`，vLLM 自动切换到 `spawn` 模式。`spawn` 模式会重新导入主模块，如果代码不在 `if __name__ == "__main__"` 保护下，就会无限递归创建子进程。

**解决方案：**

将所有代码放进函数中，用 `if __name__ == "__main__":` 保护入口：

```python
# ❌ 错误写法 — 直接在顶层调用
from vllm import LLM
llm = LLM(model="Qwen/Qwen2.5-1.5B")  # 这行会被子进程重复执行
llm.generate(...)

# ✅ 正确写法 — 用 main() 函数 + __name__ 保护
from vllm import LLM

def main():
    llm = LLM(model="Qwen/Qwen2.5-1.5B")
    llm.generate(...)

if __name__ == "__main__":
    main()
```

> **适用范围：** 所有 WSL2 环境下的 vLLM 脚本都必须这样写，包括离线推理和 API 服务器测试。

#### 坑 2：GPU 显存不足 (OOM)

**错误信息：**
```
ValueError: Free memory on device cuda:0 (10.98/12.0 GiB) on startup is less than
desired GPU memory utilization (0.92, 11.04 GiB). Decrease GPU memory utilization
or reduce GPU memory used by other processes.
```

**原因：**

vLLM 默认 `max_model_len=131072`（Qwen2.5-1.5B 支持的最大上下文长度）。这意味着要分配一个巨大的 KV cache。对于 12GB 显卡的 RTX 3060，光 KV cache 就远超可用显存。

显存分配分析：

| 组件 | 大小 |
|---|---|
| 模型权重 (Qwen2.5-1.5B) | ~3.0 GiB |
| KV cache (max_len=131072) | **数十 GiB** ← 远超 12GB |
| CUDA Graph | ~0.5 GiB |
| 桌面 (Xwayland) | ~2.7 GiB |

**解决方案：**

限制两个关键参数：

```python
llm = LLM(
    model="Qwen/Qwen2.5-1.5B",
    max_model_len=4096,           # 131072 → 4096, 大幅缩小 KV cache
    gpu_memory_utilization=0.85,  # 0.92 → 0.85, 留更多空间
)
```

调整后的显存分配：

| 组件 | 大小 |
|---|---|
| 模型权重 | ~3.0 GiB |
| KV cache (max_len=4096) | ~6.1 GiB |
| CUDA Graph | ~0.5 GiB |
| 桌面 | ~2.7 GiB |
| **合计** | **~12.3 GiB** (刚好够用) |

> **如何选择 max_model_len？** 取决于你的使用场景。短文本对话 2048 够了，长文档需要 4096+。在有限显存下，这是一个 **上下文长度 vs 并发数** 的权衡——max_model_len 越小，同样的显存能服务更多并发请求。

## 第三步：OpenAI 兼容 API 服务

### 目标

用 `vllm serve` 启动一个 OpenAI 兼容的 HTTP API 服务器，然后用 Python 客户端调用。

### 启动服务器

```bash
NO_PROXY="*" vllm serve Qwen/Qwen2.5-1.5B \
    --max-model-len 4096 \
    --gpu-memory-utilization 0.85
```

### 测试脚本 (`02_api_server.py`)

```python
"""
vLLM OpenAI 兼容 API 服务器测试

启动服务:
    NO_PROXY="*" vllm serve Qwen/Qwen2.5-1.5B --max-model-len 4096 --gpu-memory-utilization 0.85

然后用 curl 或 Python 客户端调用。
"""

import sys
import time
import httpx

BASE_URL = "http://localhost:8000"


def wait_for_server(timeout=120):
    """等服务就绪"""
    start = time.time()
    while time.time() - start < timeout:
        try:
            r = httpx.get(f"{BASE_URL}/health", timeout=10)
            if r.status_code == 200:
                print("Server is ready!")
                return True
        except (httpx.ConnectError, httpx.ConnectTimeout):
            pass
        time.sleep(2)
    return False


def test_models():
    """列出可用模型"""
    r = httpx.get(f"{BASE_URL}/v1/models", timeout=10)
    data = r.json()
    print("=== Models ===")
    for m in data["data"]:
        print(f"  {m['id']}")
    print()
    return data["data"][0]["id"]


def test_completion(model_id):
    """测试 /v1/completions"""
    print("=== Completion ===")
    r = httpx.post(
        f"{BASE_URL}/v1/completions",
        json={
            "model": model_id,
            "prompt": "San Francisco is a",
            "max_tokens": 32,
            "temperature": 0.7,
        },
        timeout=30,
    )
    data = r.json()
    print(f"  Prompt:    'San Francisco is a'")
    print(f"  Generated: {data['choices'][0]['text']!r}")
    print(f"  Tokens:    {data['usage']}")
    print()


def test_chat_completion(model_id):
    """测试 /v1/chat/completions"""
    print("=== Chat Completion ===")
    r = httpx.post(
        f"{BASE_URL}/v1/chat/completions",
        json={
            "model": model_id,
            "messages": [
                {"role": "system", "content": "You are a helpful assistant."},
                {"role": "user", "content": "What is the capital of Japan?"},
            ],
            "max_tokens": 64,
            "temperature": 0.7,
        },
        timeout=30,
    )
    data = r.json()
    msg = data["choices"][0]["message"]["content"]
    print(f"  User: What is the capital of Japan?")
    print(f"  Bot:  {msg}")
    print(f"  Tokens: {data['usage']}")
    print()


def main():
    if not wait_for_server():
        print("Server not ready, exiting")
        sys.exit(1)

    model_id = test_models()
    test_completion(model_id)
    test_chat_completion(model_id)


if __name__ == "__main__":
    main()
```

  Tokens: {'prompt_tokens': 4, 'total_tokens': 36, 'completion_tokens': 32}

=== Chat Completion ===
  Bot: I'm sorry, but the term ".IsAny"... (1.5B 模型 chat 能力有限)

### 踩坑记录

#### 坑 3：HTTP 代理拦截 localhost 请求

**现象：**

curl 和 Python 的 httpx 请求 `http://localhost:8000` 一直超时，连接不上。

**诊断过程：**

用 `curl -v` 发现请求走了代理：

```
* Uses proxy env variable http_proxy == 'http://127.0.0.1:7890'
*   Trying 127.0.0.1:7890...
> GET http://127.0.0.1:8000/health HTTP/1.1
< HTTP/1.1 502 Bad Gateway
```

代理服务器不认识 localhost 目标，返回 502。

**解决方案：**

绕过代理。对所有访问 vLLM 服务器的命令加 `NO_PROXY`：

```bash
# curl
NO_PROXY="*" curl --noproxy '*' http://127.0.0.1:8000/health

# python
NO_PROXY="*" python3 02_api_server.py
```

或者在脚本里设置环境变量：

```python
import os
os.environ["NO_PROXY"] = "*"
os.environ["no_proxy"] = "*"
```

> **为什么用 `NO_PROXY="*"`？** 简单粗暴但有效——告诉所有 HTTP 客户端不要走代理。在生产环境应该只排除 `localhost,127.0.0.1`，但学习/开发环境用 `*` 更方便。

#### 坑 4：服务器启动慢，误以为卡死

**现象：**

`vllm serve` 启动后，`curl http://localhost:8000/health` 一直超时。端口在监听（`ss -tlnp | grep 8000` 可以看到），但连接无法建立。

**原因：**

vLLM 服务器启动流程是：

1. **加载模型权重**（~5s，从本地缓存）
2. **torch.compile 编译**（~10s）
3. **CUDA Graph 捕获**（~5s）
4. **注册路由，开始接受请求**

在第 1-3 步期间，uvicorn 的事件循环被阻塞，无法处理任何 HTTP 请求——包括 `/health`。端口虽然在监听，但 TCP 连接无法完成（SYN 包到了但 accept 不了）。

**时间线（RTX 3060 上实测）：**

```
00:00  启动命令
00:09  模型权重加载完成 (5s)
00:12  torch.compile 完成 (3s，有缓存时)
00:17  CUDA Graph 捕获完成 (5s)
00:18  服务就绪，开始接受请求
```

首次运行需要额外下载模型（~95s），后续有缓存后启动约 **60-90 秒**。

**解决方案：**

不要慌，**等就好**。写脚本时设置合理的超时时间：

```python
# 至少等 120 秒
def wait_for_server(timeout=120):
    start = time.time()
    while time.time() - start < timeout:
        try:
            r = httpx.get("http://localhost:8000/health", timeout=10)
            if r.status_code == 200:
                return True
        except (httpx.ConnectError, httpx.ConnectTimeout):
            pass
        time.sleep(2)
    return False
```

> **如何区分"还在启动"和"真的卡死了"？** 看 GPU 显存占用：`nvidia-smi`。如果显存占用在增长（模型加载中）或稳定在 11-12GB（启动完成），说明还在正常工作。如果 CPU 和 GPU 都空闲但端口连不上，那才是真的卡了。

## 第四步：关键参数探索

### 目标

通过对比实验，理解 `max_model_len`、`gpu_memory_utilization`、`enforce_eager` 三个参数对显存和性能的影响。

### 测试脚本 (`03_params.py`)

测试了三组参数组合，每组用相同的 3 个 prompt、max_tokens=64：

| # | max_model_len | gpu_mem_util | enforce_eager | 说明 |
|---|---|---|---|---|
| 1 | 2048 | 0.85 | False | 短上下文，基准 |
| 2 | 4096 | 0.90 | False | 长上下文 + 更多显存 |
| 3 | 4096 | 0.85 | True | 无 CUDA Graph |

### 实测结果

| 配置 | KV Cache | 并发数 | 吞吐量 | 启动时间 |
|---|---|---|---|---|
| len=2048, mem=0.85 | 6.51 GiB (243K tokens) | **119x** | 119.8 tok/s | ~48s |
| len=4096, mem=0.90 | 6.68 GiB (250K tokens) | **61x** | **186.8 tok/s** | ~33s |
| len=4096, mem=0.85, eager | 6.58 GiB (246K tokens) | **60x** | 150.1 tok/s | **~27s** |

> 并发数 = KV cache 总 tokens ÷ max_model_len，表示同一时刻能处理多少个请求。

### 关键发现

#### 1. `max_model_len` — 上下文长度 vs 并发数

这是影响最大的参数。它决定了 KV cache 每个请求预分配的最大 token 数。

- **2048**：并发 119x，但只能处理短文本
- **4096**：并发 61x，能处理中等长度文档
- **131072**（默认）：单个请求就需要 ~20GB KV cache，12GB 卡直接 OOM

**选择原则：** 根据实际场景的最长输入+输出长度来设定。留 20-30% 余量。

#### 2. `gpu_memory_utilization` — 显存利用率

控制 vLLM 总共能使用多少 GPU 显存。

- **0.85**：在桌面运行时安全值（Xwayland 占 ~2.7GB，12 - 2.7 = 9.3GB 可用，0.85 × 12 = 10.2GB，不会溢出）
- **0.90**：KV cache 更多，吞吐更高，但留给系统的余量更少
- **0.92+**（默认）：在桌面运行时会 OOM

**注意：** 这个比例是相对**总显存**的，不是相对可用显存。12GB × 0.85 = 10.2GB。

#### 3. `enforce_eager` — 启动速度 vs 推理速度

| | CUDA Graph (默认) | enforce_eager=True |
|---|---|---|
| 启动时间 | 60-90s | 20-30s |
| 推理吞吐 | 186.8 tok/s | 150.1 tok/s |
| 显存占用 | 多 ~0.5 GiB (CUDA Graph) | 少 ~0.5 GiB |

**适用场景：**
- **开发/调试**：用 `enforce_eager=True`，快速迭代
- **生产/压测**：用默认值，CUDA Graph 带来 ~20% 性能提升
