## 图1：RadixAttention 交互演示（SGLang）

### 这张图要回答的问题
> "SGLang 的 Radix Tree 是如何随着请求一棵棵长出来的？每次新请求来，树的哪个部分被复用，哪个部分是新增的？"

### 界面布局
- **左侧**：6 个请求按钮（请求①到⑥）+ 重置按钮
- **右侧上方**：Radix Tree 可视化（树会随操作动态变化）
- **右侧下方**：信息框（显示这次请求命中了多少 token，以及原因）

### 操作步骤（建议顺序点击）

**第 1 步：点击"请求①：系统提示 + 用户A问候"**
- 树从空变成两个节点：`[SYS]你是助手` → `用户:你好 助手:嗨！`
- **意义**：首次请求，缓存为空，所有内容重新计算后存入树中。绿色节点 = 新插入。

**第 2 步：点击"请求②：用户A 第2轮"**
- 前两个节点变蓝（命中），新增第三个绿色节点 `用户:天气 助手:晴`
- 信息框显示：命中 3/5 个 token 的 KV Cache，节省 60% prefill
- **意义**：这就是 RadixAttention 的核心——相同前缀在树上找到了，直接复用，跳过计算。蓝色 = 命中复用。

**第 3 步：点击"请求③：用户B 新对话"**
- 树在 `[SYS]你是助手` 处分叉，右边长出新分支 `用户:再见 助手:拜拜`
- **意义**：用户B 和用户A 共享同一个系统提示节点（最长公共前缀），分叉后各走各的分支。这叫"自然共享"，无需手动配置。

**第 4 步：点击"请求④：Few-shot 学习"**
- 树根部又分叉出一条全新分支（无公共前缀），独立建树
- **意义**：与之前的对话完全不同的内容，从根部另建分支。树是压缩前缀树，只存公共部分。

**第 5 步：点击"请求⑤：同组Few-shot，不同问题"**
- Few-shot 分支上，`例子1:猫→喵` 和 `例子2:狗→汪` 变蓝（命中），末尾分叉出新叶 `问:鱼→???`
- 命中 2/3 个节点
- **意义**：两条 Few-shot 请求共享前两个示例（公共前缀），只有最后一个问题不同，极大节省计算。

**第 6 步：点击"请求⑥：用户A 第3轮"**
- 用户A 的整条历史链（5个节点）全部变蓝，新增第6轮叶子
- 命中率 5/6，节省 83% prefill
- **意义**：多轮对话越长，历史前缀越长，命中率越高，TTFT 越低。这是 SGLang 在多轮场景的核心优势。

### 颜色含义速查
| 颜色 | 含义 |
|------|------|
| 灰色 | 已缓存，当前请求未涉及 |
| 绿色 | 本次新插入的节点 |
| 蓝色 | 本次命中并复用的节点 |

### 点击"🔄 重置"后
树清空回到初始状态，可以重新从第1步开始，加深理解。

## 图2：Compressed FSM 交互演示（SGLang）

### 这张图要回答的问题
> "生成 JSON 时，为什么有些 token 不需要一个个算？Compressed FSM 到底压缩了什么？"

### 界面布局
- **左侧上方**：正在生成的 JSON 输出（token 序列，颜色代表状态）
- **左侧下方**：统计框（Forward Pass 次数）+ 两个操作按钮
- **右侧上方**：旧系统普通 FSM（状态图）
- **右侧下方**：新系统 Compressed FSM（压缩后的状态图）
- **底部**：两种方案对比表

### 操作步骤

**方案A：点击"旧系统 下一步"（多次点击）**

每点一次，模拟旧系统执行一次 Forward Pass，只生成一个 token。

- 点第1次：生成 `{`，Forward Pass 计数 +1
- 点第2次：生成 `"`，Forward Pass 计数 +1
- 点第3~5次：继续生成 `summary`、`"`、`:`、`_"`
- **观察**：即使 `{"summary": "` 这段字符串在 JSON schema 下唯一确定（只有一个合法选项），旧系统仍然每个 token 跑一次完整的 GPU Forward Pass

**方案B：点击"新系统 下一步"**

- 点第1次：一次性跳过整段 `{"summary": "`（7个确定性 token），Forward Pass 仅 +1
- 点第2次：开始自由生成内容（模型决定写什么），每个 token +1
- 点第3次：一次性输出结尾 `"}`，Forward Pass +1
- **观察**：整个 JSON 生成只用了 2~3 次 Forward Pass，而旧系统需要 7+N 次

**点击"↺"重置**，可以切换对比：先用旧系统点完，再重置用新系统，直观看到 Forward Pass 次数的差距。

### token 颜色含义
| 颜色 | 含义 |
|------|------|
| 暗色（未生成）| 还没到这一步 |
| 紫色（闪烁）| 当前正在生成的 token |
| 绿色 | 确定性 token（JSON 结构部分，被压缩跳过） |
| 红色 | 自由生成 token（模型采样） |

### 关键理解点
JSON 中的结构字符（`{`、`"`、`:`、`}`、键名）在 schema 约束下只有**唯一**合法选项，
Compressed FSM 识别出这段"单选路径"后，直接一步输出整段，避免了无谓的 Forward Pass。
只有模型自由生成的内容部分（字段值）才需要逐 token 采样。

## 图4：PagedAttention 交互演示（vLLM）

### 这张图要回答的问题
> "传统方案为什么浪费这么多显存？PagedAttention 怎么解决的？Copy-on-Write 是怎么回事？"

### 界面布局（三个标签页）

---

### 标签页①：传统方案（预分配）

**操作步骤：**

1. **点击"➕ 加入新序列"（第1次）**
   - 物理显存上出现一块连续的蓝色区域（8块）
   - 蓝色 = 序列A实际使用，红色 = 预分配但浪费
   - 利用率数字出现在右下角
   - **意义**：序列A实际生成了3块（48 token），但系统预先占了8块（128 token）的连续空间，尾部5块浪费

2. **点击"➕ 加入新序列"（第2次）**
   - 序列B进来，继续找连续8块
   - **意义**：序列B完成得更快（只用2块），剩余6块全部浪费

3. **点击"➕ 加入新序列"（第3次）**
   - 如果成功：序列C进来，利用率更低
   - 如果显示红色警告"外部碎片导致加入失败"：**这是重点！** 总显存还有空闲块，但因为不连续，新序列放不进去
   - **意义**：外部碎片问题——空间够用但装不下

**观察指标：**
- "已占块" = 被序列占用的总块数
- "浪费块" = 预分配但空着的块
- "利用率" = (已占-浪费)/已占，往往只有 20-40%

---

### 标签页②：PagedAttention（分块）

**操作步骤：**

1. **点击"➕ 新序列加入"**
   - 右侧 Block Table 出现第一行：序列A 逻辑块0 → 物理块#某处
   - 物理块随机散落在显存各处（不要求连续）
   - **意义**：第一个块，按需分配，无需预留

2. **多次点击"▶ 生成一个token"**
   - Block Table 中当前块的"token数"从1涨到16
   - 涨到16时，下一次点击会自动申请新物理块
   - **意义**：每 16 个 token 才申请一个新块，精确按需，无内部浪费

3. **点击"➕ 新序列加入"（第2条）**
   - Block Table 新增第2条序列，两条序列的物理块互不相关
   - **意义**：序列间完全独立，Block Table 负责映射，物理块可以在任意位置

4. **继续点击"▶ 生成一个token"**
   - 系统随机选一条序列生成，观察两条序列如何同时向前推进
   - **意义**：PagedAttention 下多序列共存，利用率接近 100%（只有各自尾块有少量浪费）

### 标签页③：Copy-on-Write 共享

**操作**：直接阅读，无需交互。

图示展示 Prompt 块被3条输出路径共享的情况（beam search 或 parallel sampling）：
- 共同的 Prompt 部分：3条路径共享同3个物理块，引用计数=3
- 各自生成的部分：每条路径独自持有自己的块

**关键点**：只要引用计数>1，任何路径想修改时必须先复制（写时复制）。Prompt 部分只读，永远不需要复制。

## 图5：Continuous Batching 交互演示（vLLM）

### 这张图要回答的问题
> "静态批处理为什么让 GPU 空转？Continuous Batching 是如何在 iteration 级别塞入新请求的？调度器的三个状态怎么流转？"

### 界面布局（三个标签页）

---

### 标签页①：两种方案对比

**操作**：直接阅读两张时序图，对比 GPU 利用率。

左图（静态）：
- Seq B、C 提前完成，但必须等 Seq A（最长序列）才能释放
- 等待期间 GPU 完全空转（虚线部分）
- 新请求 D 在队列里干等整批结束

右图（Continuous Batching）：
- Seq B 完成 → D 立刻顶上，无需等待
- Seq C 完成 → E 立刻顶上
- GPU 利用率接近 100%

---

### 标签页②：调度器模拟（核心交互）

**操作步骤（建议操作流程）：**

1. **点3次"➕ 新请求到达"**
   - WAITING 队列出现3个彩色请求卡片（A、B、C）
   - RUNNING 批次立刻被填满（最大并发=3）
   - **意义**：有空位就立刻从 WAITING 移入 RUNNING

2. **点击"▶ 执行一次 iteration"**
   - 每个运行中请求的"剩余步数"减1
   - 如果某个请求减到0，卡片消失（完成）
   - 右侧日志记录：谁完成了（绿色 ✓），谁补了进来（黄色 ↑）
   - **意义**：一次 iteration = GPU 执行一次前向传播，为批次中所有序列各生成一个 token

3. **再点"➕ 新请求到达"，然后立刻点"▶ 执行一次 iteration"**
   - 观察：新请求加入 WAITING 队列，下一次 iteration 完成后有序列退出，新请求立刻补入
   - **意义**：这就是 Continuous Batching——无缝衔接，不等待

4. **观察"GPU利用率"指标**
   - RUNNING 有 3 条 = 100%，2 条 = 67%，1 条 = 33%，0 条 = 0%
   - **意义**：只要 WAITING 队列有请求，利用率就趋近 100%

---

### 标签页③：三状态机

**操作**：阅读状态图，理解状态流转。

三个状态：
- **WAITING**：请求到达，等待分配显存
- **RUNNING**：正在 GPU 上处理，每个 iteration 生成一个 token
- **PREEMPTED**：被抢占，KV Cache 被丢弃或换出到 CPU

关键流转：
- WAITING → RUNNING：有空闲 KV Cache 块，可以开始处理
- RUNNING → PREEMPTED：新请求涌入导致显存不足，当前序列被迫暂停
- PREEMPTED → WAITING（recompute 策略）：丢弃 KV Cache，重新排队，下次从头计算前缀
- PREEMPTED ↔ RUNNING（swap 策略）：KV Cache 搬到 CPU，恢复时再搬回，代价是 PCIe 带宽

## 图6：APC 自动前缀缓存（vLLM）

### 这张图要回答的问题
> "Chain Hash 怎么工作的？缓存命中时究竟跳过了什么？vLLM APC 和 SGLang 的区别在哪里？"

### 界面布局（三个标签页）

---

### 标签页①：工作原理

**操作**：阅读 Chain Hash 示意图，理解"哈希链"的传递规则。

关键规则：
- Block#0 的哈希 = hash(Block#0 的 16 个 token 内容) = `a1b2`
- Block#1 的哈希 = hash(Block#0 的哈希 `a1b2` + Block#1 的 token 内容) = `c3d4`
- 以此类推，每个块的哈希都"记住"了它之前所有块的内容
- 任意一个 token 改变 → 后面所有块的哈希全部改变 → 旧缓存全部失效（自动保证正确性）

图中请求② 和请求① 前两个块哈希完全相同 → 直接复用，跳过这 32 token 的 prefill 计算。

---

### 标签页②：缓存模拟（核心交互）

**操作步骤（建议按顺序点击）：**

1. **点击"请求A：长系统提示 + 问题1"**
   - 请求块序列全部显示红色（✗ 重算）
   - 缓存槽从空变成3个蓝色系统提示块 + 1个问题块
   - **意义**：首次，缓存为空，全部重算，但结果存入缓存

2. **点击"请求B：长系统提示 + 问题2"**
   - 前3个块（系统提示）变绿（✓ 命中），最后1块（问题2）红色（✗ 重算）
   - 统计：命中3块，节省 48 token 的 prefill
   - **意义**：这是 APC 的主要使用场景——大量请求共享相同系统提示，只算一次，后续全部复用

3. **点击"请求C：短系统提示 + 问题3"**
   - 只有第1块命中（短系统提示和长系统提示共享前16 token），后面未命中
   - **意义**：系统提示越长，共享的块越多，收益越大

4. **点击"请求D：无系统提示"**
   - 全部未命中（红色）
   - **意义**：没有共享前缀 = APC 无法起效

5. **点击"🗑 清空缓存(LRU驱逐)"**
   - 缓存槽清空
   - **意义**：模拟 LRU 驱逐后的冷启动，重新发送请求观察效果

### 标签页③：vLLM的局限 vs SGLang

**操作**：阅读对比图。

**核心差距**（图中展示 18-token 前缀的情况）：
- vLLM：命中 Block#0（16 token），Block#1 只有 2 token 凑不成完整块，整块失效，重新计算
- SGLang：18 token 全部命中，因为 Radix Tree 以单 token 为索引，不受块边界限制

**通俗类比**：
- vLLM 像"只能整张纸复印"——18页文档，只能复用前16页，最后2页手抄
- SGLang 像"可以按页复印"——18页全复用，一页都不浪费

## 图7：KV Cache 架构对比图（vLLM vs SGLang）

### 这张图要回答的问题
> "把两个框架放在一起看，同一个输入下行为有什么不同？数据结构的根本差异是什么？"

### 界面布局（四个标签页）

---

### 标签页①：场景对比（最重要）

**四个场景，建议全部点一遍：**

**场景①：首次请求**
- 两框架都全部未命中（缓存为空）
- **意义**：起点相同，无差异

**场景②：相同系统提示（第二条请求）**
- vLLM：Block#0、Block#1 命中（hash 一致），命中率 66%
- SGLang：系统提示节点（32 token）全部命中，命中率 66%
- **意义**：系统提示恰好是整块对齐时，两者命中率相同。这说明差距不是"有没有缓存"，而是"粒度"

**场景③：18-token 非整块前缀（最关键）**
- vLLM：命中 1/3 块（33%），18 token 中只复用了 16 token
- SGLang：命中 100%，18 token 全部复用
- **意义**：这是两者最核心的差距所在，实际场景中消息长度很少恰好是 16 的整数倍

**场景④：多轮对话第3轮**
- vLLM：如果历史恰好对齐，命中 3/4 块；如有余数则更低
- SGLang：历史部分（无论长度）全部命中，只算第3轮新输入
- **意义**：多轮场景中每轮消息长度随机，SGLang 的优势随轮数单调增长

---

### 标签页②：粒度对比（拖动滑块）

**操作**：拖动"前缀长度"滑块（1~48 token）

- **滑到 16、32、48**：两者命中率相同（整块对齐）
- **滑到 17、18、20 等非整数倍**：vLLM 命中率下降，SGLang 始终 100%
- **看底部信息框**：显示具体的差距量（损失了几个 token）

**结论**：前缀长度是 16 的整数倍时两者等价；其他情况 SGLang 必然更优。

---

### 标签页③：数据结构

**操作**：阅读两张内部结构图。

**vLLM（左）**：
- 逻辑层（序列）→ Block Table（CPU，哈希表）→ 物理块（GPU，非连续）
- Chain Hash 在块填满时计算，存 CPU

**SGLang（右）**：
- 所有请求共用一棵 Radix Tree（CPU）
- 树节点对应 KV 张量（GPU）
- lock_ref 防止正在使用的节点被驱逐
