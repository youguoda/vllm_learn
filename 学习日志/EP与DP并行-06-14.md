# Expert Parallelism 与 Data Parallelism

> 单卡读文档 + 纸面推演。搞懂 MoE 模型为什么用 EP（Expert Parallel）而非 TP，以及 SGLang 招牌的 DP Attention。

---

## 步骤 1：MoE 路由与 EP 的直觉

```
普通 Transformer FFN:
  每个 token → 同一个 FFN → 输出
  TP 切分: 把这个 FFN 的权重按列/行切到多卡, 每个 token 过所有卡(All-Reduce)

MoE Transformer FFN (DeepSeek-V3 / Mixtral):
  每个 token → Router → 选 K 个专家 → 各自计算 → 加权合并
  EP 切分: N 个专家分到 N 张卡, 每卡只存一部分专家
           每个 token 只去它激活的 K 个专家所在的卡(All-to-All)
```

**核心通信原语差异**：

| 并行 | 原语 | 含义 |
|---|---|---|
| TP | **All-Reduce** | 每卡都要完整结果（广播式求和） |
| EP | **All-to-All** | token 定向去激活的专家所在卡（点对点路由） |

> SGLang 用 **DeepEP** 后端做 MoE 的 All-to-All（源码 `srt/eplb/`，还有 EPLB 专家负载均衡器 `expert_distribution.py`）。`--expert-parallel-size`/`--ep-size` 开启。

---

## 步骤 2：DeepSeek-V3 EP 通信量手算

```
DeepSeek-V3 MoE: 总专家 256, 每 token 激活 8, hidden=7168, fp16(2字节)
场景: EP=8, 每卡 256/8=32 个专家

每个 token 路由:
  激活 8 专家, 理想均匀分布在 8 卡 → 每卡平均收 1 个 token 的 hidden state
  hidden state 大小 = 7168 × 2 = 14 KB
  All-to-All 发送 + 接收(结果返回)

batch=32 每步 All-to-All(理想均匀):
  ≈ 32 tokens × 14KB × (激活卡/EP) ≈ 32 × 14KB ≈ 448 KB (下界)
```

### TP vs EP 通信对比

| 并行 | 原语 | 通信量(batch=32) | 适用 |
|---|---|---|---|
| TP=8 | All-Reduce | ~2.5 MB/层 | Dense 模型，单机 |
| EP=8 | All-to-All | ~448 KB/层（均匀路由） | MoE，专家多 |

> **关键洞察**：MoE 用 EP 比 TP 省通信——每个 token 只去**少数激活的专家**，不需要广播给所有卡。TP 把同一层权重切开，每个 token 前向必过所有卡（All-Reduce 收集）；EP 把不同专家放不同卡，token 定向路由（All-to-All）。
>
> 注意：EP 的实际通信量高度依赖**路由分布均匀性**——若热门专家集中在少数卡，会造成负载不均（这就是 EPLB 专家负载均衡器要解决的）。448KB 是理想均匀下界。

## 步骤 3：SGLang DP Attention（招牌设计）

`--enable-dp-attention`：SGLang 的特色。背景——MoE 模型里 Attention 层和 FFN 层**并行策略可以不同**：

```
MoE 模型分层并行:
  FFN 层  → EP (专家分卡, All-to-All)
  Attn 层 → DP (每卡独立处理一部分请求, KV cache 不切分)
```

- **好处**：Attention 的 KV cache 不跨卡 → 不需要 All-Reduce/All-Gather，减少通信。
- **代价**：每卡要存**完整的 Attention 权重**（不切分）。

> **适用条件**：batch 很大、MoE 的 FFN 已经用 EP 分卡、Attention 不是瓶颈时——DP Attention 更简单高效。这是 SGLang 针对 DeepSeek-V3 这类大 MoE 的专门优化。

SGLang 还有 `--enable-dp-lm-head`、`--enable-two-batch-overlap` 等配套 DP 优化。

---

## 步骤 4：vLLM 的对应方案

vLLM 也支持 data-parallel/expert-parallel（通过 ParallelConfig），但**没有 SGLang 那种单进程内 Attention/FFN 异构并行的 `--enable-dp-attention` 命名特性**。多实例扩展通常靠外部负载均衡。

```
SGLang 方案:                    vLLM 方案:
  单进程内 DP Attention            多进程 + 外部 LB
  优势: 低延迟, 单引擎管理          优势: 实现简单, 水平扩展灵活
  劣势: 配置复杂                    劣势: LB 额外延迟, KV 不跨实例共享
```

> **架构选择哲学差异**（分享可讲）：SGLang 倾向"单引擎内做精细的异构并行"（DP Attention + EP FFN 混搭），追求极致；vLLM 倾向"单引擎做好，扩展交给外部编排"，追求简单可组合。这呼应了 [[KVCache设计哲学-HiCache-06-14]] 里"SGLang 心疼每一份计算/SGLang 做到极致 vs vLLM 简单可靠"的一贯调性。

## 步骤 5：自测

**Q：为什么 MoE 用 EP 比 TP 省通信？**

> **答**：TP 是把**同一层的权重切开**，每个 token 的前向必须经过所有卡，用 All-Reduce 收集结果——通信量 ∝ batch × hidden（全量广播求和）。
>
> EP 是把**不同专家放不同卡**，每个 token 只去它激活的 K 个专家所在的卡（All-to-All 定向路由）——通信量 ∝ token 数 × hidden × (激活专家/总专家比例)。因为 MoE 每个 token 只激活少数专家（如 256 选 8），EP 的定向路由远小于 TP 的全量广播。
>
> 一句话：**TP 广播给所有卡，EP 只发给该去的卡**。MoE 的稀疏激活特性让 EP 的"定向"优势凸显。

## 一句话

> **TP 切权重(All-Reduce 广播), EP 分专家(All-to-All 定向), DP 复制处理不同请求。** MoE 模型的稀疏激活让 EP 比 TP 省通信; SGLang 的 DP Attention 是"FFN 用 EP + Attn 用 DP"的异构并行招牌, vLLM 走"单引擎+外部 LB"的简单路线。
