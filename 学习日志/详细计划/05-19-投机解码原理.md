## 步骤 1：先制造"动机"（30 min）

亲眼看 decode 阶段 GPU 在摸鱼，再去理解投机解码要解决什么：

```bash
# 启动 server
HF_HUB_OFFLINE=1 NO_PROXY="*" vllm serve Qwen/Qwen2.5-1.5B \
  --max-model-len 4096 --gpu-memory-utilization 0.85

# 另一个终端：跑单请求长输出，同时用 nvidia-smi 盯 GPU 利用率
watch -n 0.5 "nvidia-smi --query-gpu=utilization.gpu,utilization.memory \
  --format=csv,noheader,nounits"
```

```python
# single_long_decode.py — 单请求、长输出，让 decode 成为主要阶段
import time, openai
client = openai.OpenAI(base_url="http://localhost:8000/v1", api_key="x")

if __name__ == "__main__":
    t0 = time.time()
    r = client.chat.completions.create(
        model="Qwen/Qwen2.5-1.5B",
        messages=[{"role":"user","content":"写一篇2000字的工业4.0发展报告。"}],
        max_tokens=600, stream=False)
    print(f"耗时: {time.time()-t0:.1f}s，输出 {len(r.choices[0].message.content)} 字")
```

记录 GPU 利用率（大概率 30-50%，远未饱和）。这就是投机解码的动机：**decode 每步只生成 1 个 token，GPU 的大量并行算力在等下一步的串行依赖，白白浪费了。**

## 步骤 2：理解 draft-verify 范式（50 min）

核心思路：用一个**小模型**（或简单规则）先猜 K 个 token，再用**大模型一次并行验证**。

```
传统 decode（每步串行）：
  大模型(t1) → 输出 token_1
  大模型(t2) → 输出 token_2  ← 要等 token_1 结束
  大模型(t3) → 输出 token_3  ← 要等 token_2 结束
  共 3 次前向，3 步延迟

投机解码（draft+verify）：
  小模型猜测: draft = [token_1?, token_2?, token_3?]  ← 极快（小模型 3 步）
  大模型验证: verify([token_1?, token_2?, token_3?])  ← 1次前向，并行验证3个位置
    → 大模型在每个位置输出其认为正确的 token
    → 如果前 k 个猜对了，接受，接着猜
    → 第一个猜错的位置后全丢弃，从那里重新猜

关键保证：最终输出分布与直接用大模型完全相同（通过"推测采样"算法保证）
```

写进笔记：投机解码省的不是"每步计算量"，而是**减少了串行步数**（接受 K 个 draft = 做了 K 步 decode 却只花了约 1 步的时间），代价是验证步骤比正常 decode 略重（要同时算 K+1 个位置）。

## 步骤 3：读 EAGLE 论文核心思路（40 min）

arXiv:2401.15077，只读摘要 + §3 方法：

- **普通 draft 模型方案**：用一个小模型（如 7B 用 1B draft）独立生成 draft，问题是小模型和大模型的特征空间不对齐，接受率不高
- **EAGLE 的改进**：不训练独立小模型，而是训练一个"特征级 draft head"——输入大模型最后一层的**隐状态**（特征向量），而不是 token embedding，预测下一步的隐状态，再用 LM head 解码
- **效果**：因为 draft head 直接复用大模型的特征，接受率更高（~80%），加速比更稳定

Medusa（arXiv:2401.10774）的对比：不用 draft 模型，而是在大模型顶部加多个平行解码头（head_1 预测 t+1，head_2 预测 t+2 …），一次前向同时猜多步，再用树状验证。

## 步骤 4：画对比图（20 min）

纸面画出三种方案的时序对比：

```
普通 decode：    [大] [大] [大] [大] [大]  —— 串行，每步独立

Draft+Verify：  [小][小][小] [大-验证3] [小][小][小] [大-验证] ...
                 ↑猜3步      ↑1次验证    ↑继续猜

EAGLE：          [大-特征] → [draft head×3] → [大-验证3+1]
                 ↑复用特征，draft质量更高

Medusa：         [大 + head1 + head2 + head3] → 树状验证
                 ↑一次前向同时猜多步，无draft模型
```

自测：为什么验证 K 个 draft token 的成本 ≈ 生成 1 个 token？因为验证是把 K 个位置**并行**丢进 Transformer（attention 是并行的），计算量主要取决于 batch size（K+1 个 token），而不是 K 次串行前向。
