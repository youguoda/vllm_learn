# 06-08 详细计划：社区数据交叉验证

## 步骤 1：定向收集（50 min）

针对报告里信心 <4 星的结论，找社区数据支撑或对立证据：

**① 官方最新 benchmark**

```
vLLM benchmark blog:
  https://blog.vllm.ai → 搜 "benchmark" / "throughput"

SGLang benchmark:
  https://lmsys.org/blog → 搜 "SGLang" + "throughput" / "latency"
  https://github.com/sgl-project/sglang → 查 README 里的 benchmark 表格
```

摘录格式：

```markdown
### 来源1：[标题]（[链接]，[日期]）
- 测试条件：模型=___, 硬件=___, 框架版本=___
- 关键数据：___
- 与本实验的关系：[一致/矛盾/无法比较]，原因：___
```

**② 高质量第三方评测**

搜索关键词：
- `vllm sglang benchmark comparison 2025`
- `site:github.com vllm sglang throughput`
- `huggingface blog inference benchmark`

筛选标准：有明确硬件型号 + 模型 + 版本号，数据可溯源。

**③ 针对性查找"矛盾证据"**

主动搜索和你结论相反的观点：
- "SGLang 不比 vLLM 快" 类帖子 → 看测试条件，是否低复用负载（和你结论一致）
- "vLLM 多轮对话更好" → 看是否是 APC 已经够用的场景

## 我的实验 vs 社区的主要差异来源

1. **模型规模**：我用 1.5B，社区常用 7B/13B/70B
   - 影响：小模型 decode 极快，瓶颈在 prefill；大模型反之
   - 外推原则：趋势可外推，绝对数字不可外推

2. **GPU 代际**：我用 Ampere 3060，社区常用 A100/H100
   - 影响：带宽差异影响 decode 速度；H100 有 FP8 计算加速
   - 外推原则：相对比较（百分比差异）可外推

3. **版本差异**：两框架更新很快
   - 影响：某些版本 bug 或优化会改变结论
   - 标注：报告中注明版本号，读者自行对应

## 需要在报告中明确标注的项目

- [ ] "多轮对话 SGLang TTFT 快 __%" 加注：基于 1.5B + 3060，趋势可外推
- [ ] "通用负载持平" 加注：差异 <___% 在噪声范围内，大模型需重新测试
- [ ] EP/DP 并行结论加注：理论推导，未实测，参考 SGLang 官方文档
