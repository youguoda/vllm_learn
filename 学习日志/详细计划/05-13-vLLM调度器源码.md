## 步骤 2：读调度器主循环（60 min）

找 `schedule()` 方法，边读边在笔记里画状态机：

```
请求状态流转：
WAITING ──(有资源)──► RUNNING ──(EOS)──► FINISHED
                         │
                    (显存不足，被抢占)
                         │
               ┌─────────┴──────────┐
           SWAPPED                PREEMPTED
        (KV 换到 CPU)           (KV 直接丢弃，重新排队)
               │                    │
           (CPU 换回)            (重新 prefill)
               └─────────┬──────────┘
                       WAITING
```

重点读三段代码：

**① waiting → running（调度决策）**

```python

    # 检查 token budget：本轮最多能 prefill 多少 token？
    # 检查 KV 块：能分配到足够的物理块吗？
    # 通过检查 → 移入 running
```

找到 `token_budget` 或 `num_new_tokens` 相关的限制逻辑，这就是 Continuous Batching 的核心——**每个调度步 token 数有上限，保证 decode 请求不被 prefill 饿死**。

**② running 内的 token 预算分配**

找 `SchedulingBudget` 类或类似结构，理解：每步最多 prefill X token + decode Y 请求并存，两者共享 token budget。这就是为什么 long prefill 会拖慢同批次 decode 请求的 TTFT（Chunked Prefill 就是为了解决这个）。

**③ 抢占逻辑（preemption）**

```python
# 找 _preempt 或 preempt_sequence 方法：
# 策略1: RECOMPUTE — 丢弃 KV blocks，请求重回 WAITING，下次重新 prefill
# 策略2: SWAP      — KV blocks 搬到 CPU，等 CPU 内存换回再继续
```

找到选择两种策略的条件（通常：CPU 内存够 → SWAP，否则 → RECOMPUTE）。

  --gpu-memory-utilization 0.70 \

```python
# trigger_preempt.py — 同时提交多个长输出请求，争夺有限 KV 块
import asyncio, openai

client = openai.AsyncOpenAI(base_url="http://localhost:8000/v1", api_key="x")
LONG_PROMPT = "请写一篇关于工业传感器发展历史的详细报告，要求包含：起源、主要技术演进、代表性产品、未来趋势，每个章节不少于500字。"

async def req(i):
    print(f"发出请求 {i}")
    r = await client.chat.completions.create(
        model="Qwen/Qwen2.5-1.5B",
        messages=[{"role":"user","content":LONG_PROMPT}],
        max_tokens=512)
    print(f"完成请求 {i}，输出 {len(r.choices[0].message.content)} 字")

if __name__ == "__main__":
    asyncio.run(asyncio.gather(*[req(i) for i in range(8)]))
```

在 server 日志里找 `Preempted` 或 `swap` 字样，记录：抢占了第几个请求、用的是 recompute 还是 swap。
