# 05-16 详细计划：Expert Parallelism / Data Parallelism

## 步骤 1：读 SGLang EP 文档，理解 MoE 路由（50 min）

先建立直觉：MoE 模型（如 DeepSeek-V3、Mixtral）每一层不是一个 FFN，而是 N 个 FFN（专家），每个 token 只激活其中 K 个。

```
普通 Transformer FFN：
  每个 token → 同一个 FFN → 输出
  TP 切分：把这个 FFN 的权重按列/行切到多卡

MoE Transformer FFN：
  每个 token → Router → 选 K 个专家 → 各自计算 → 加权合并
  EP 切分：把 N 个专家分配到 N/tp 张卡，每卡只存一部分专家
```

读 SGLang DeepEP 文档/注释：
```bash
source ~/venv-sglang/bin/activate
SGLANG=$(python3 -c "import sglang; print(sglang.__file__.replace('/__init__.py',''))")
find $SGLANG -name "*.py" | xargs grep -l "expert_parallel\|MoE\|all_to_all" 2>/dev/null | head -5
```

重点理解 All-to-All 通信：**TP 用 All-Reduce（每卡都要结果），EP 用 All-to-All（token 只去激活的专家所在的卡，不是广播给所有卡）**。

## 步骤 2：手算 EP 通信量（40 min）

用 DeepSeek-V3 的实际参数做纸面推演（不跑代码，算完写进笔记）：

```
DeepSeek-V3 MoE 参数：
  总专家数: 256
  每 token 激活: 8 个专家
  hidden_size: 7168
  dtype: fp16（2字节）

场景: EP=8（8张卡），每卡 256/8=32 个专家

每个 token 的路由：
  - 激活 8 个专家，分布在不同卡上（平均 8/8=1 卡/token，最坏 8 卡）
  - All-to-All 发送：token 的 hidden state (7168×2=14KB) 发给各目标卡
  - All-to-All 接收：计算结果从专家卡返回

batch=32 时，每步 All-to-All 通信量估算：
  发送 = 32 tokens × 14KB × 激活卡数/EP ≈ 32 × 14KB × (8/8) = 448 KB
  （实际因路由分布而变，这是理想均匀分布下的下界）
```

对比 TP=8 的同场景（05-15 的公式）：

| 并行方式 | 通信原语 | 通信量（batch=32） | 适用场景 |
|---|---|---|---|
| TP=8 | All-Reduce | ~2.5 MB/层 | Dense 模型，单机 |
| EP=8 | All-to-All | ~448 KB/层（均匀路由） | MoE 模型，专家多 |

**关键洞察**：MoE 模型用 EP 比 TP 省通信，因为每个 token 只去少数专家，不需要广播给所有卡。

## 步骤 3：理解 SGLang DP Attention（30 min）

SGLang 独有的一个设计：`--dp-attention`（Data Parallel Attention）。

背景：MoE 模型里 Attention 层和 FFN 层并行策略可以不同：
- FFN 层用 EP（专家分卡）
- Attention 层可以用 DP（每卡独立处理一部分请求，KV cache 不切分）

好处：Attention 的 KV cache 不用跨卡通信，减少 All-Reduce 次数。代价：每卡要存完整的 Attention 权重。

```bash
# 读 SGLang 的 dp attention 选项说明
grep -rn "dp.attention\|dp_attention\|data_parallel_attn" $SGLANG --include="*.py" | head -10
```

写进笔记：DP Attention 的适用条件——batch 很大、MoE 专家层 EP 已经分卡、Attention 不成瓶颈时，DP 更简单。

## 步骤 4：vLLM 的对应方案（20 min）

vLLM 没有内置 DP Attention，多实例通常靠外部负载均衡（LB 层分发请求到多个 vLLM 实例）。

```
SGLang 方案：           vLLM 方案：
  单进程内 DP Attention    多进程 + 外部 LB
  优势：低延迟，共享 KV    优势：实现简单，水平扩展
  劣势：单点故障           劣势：LB 有额外延迟，KV 不共享
```

写进笔记这个对比，这是分享里可以讲的"架构选择哲学"差异。

## 步骤 5：自测（10 min）

回答：为什么 MoE 用 EP 比 TP 省通信？

要点：TP 是把同一层的权重切开，每个 token 的前向传播必须经过所有卡（All-Reduce 收集结果）；EP 是把不同专家放不同卡，每个 token 只去自己激活的专家所在的卡（All-to-All 定向路由），通信量 = token 数 × hidden × 激活专家比例，远小于广播全部。
