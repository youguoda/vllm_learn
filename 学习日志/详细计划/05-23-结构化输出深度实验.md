# 05-23 详细计划：结构化输出深度对比实验

## 步骤 1：FSM 编译耗时实验（40 min）

新建 `11_structured_output_deep.py`：

```python
"""
结构化输出深度对比实验
测三件事：
1. FSM 编译耗时（首次 vs 后续）
2. schema 复杂度对速度的影响
3. SGLang Compressed FSM 在"确定性字符多"的 schema 下是否更快
"""
import time, json, statistics
import openai

client_vllm   = openai.OpenAI(base_url="http://localhost:8000/v1",  api_key="x")
client_sglang = openai.OpenAI(base_url="http://localhost:30000/v1", api_key="x")
MODEL = "Qwen/Qwen2.5-1.5B"

SCHEMAS = {
    "simple": {   # 2个字段，少状态
        "type":"object",
        "properties":{
            "defect_type":{"type":"string"},
            "severity":{"type":"integer","minimum":1,"maximum":5}
        },
        "required":["defect_type","severity"]
    },
    "medium": {   # 5个字段 + enum
        "type":"object",
        "properties":{
            "id":{"type":"string","pattern":"^[A-Z]{2}[0-9]{4}$"},
            "type":{"type":"string","enum":["剥落","裂纹","磨损","腐蚀","其他"]},
            "severity":{"type":"integer","minimum":1,"maximum":5},
            "confirmed":{"type":"boolean"},
            "notes":{"type":"string"}
        },
        "required":["id","type","severity","confirmed"]
    },
    "complex": {  # 嵌套 + 数组，多状态
        "type":"object",
        "properties":{
            "report_id":{"type":"string"},
            "defects":{
                "type":"array","minItems":1,"maxItems":3,
                "items":{
                    "type":"object",
                    "properties":{
                        "type":{"type":"string","enum":["剥落","裂纹","磨损","腐蚀"]},
                        "severity":{"type":"integer","minimum":1,"maximum":5},
                        "location":{"type":"object",
                            "properties":{
                                "component":{"type":"string"},
                                "position_mm":{"type":"number"}
                            },"required":["component"]}
                    },"required":["type","severity"]
                }
            },
            "need_shutdown":{"type":"boolean"}
        },
        "required":["report_id","defects","need_shutdown"]
    }
}

PROMPT = "检测报告：传感器S042检测到轴承外圈出现明显剥落，振动幅值超标4倍，需立即处理。请生成结构化报告。"

def single_request(client, schema_name, repeat_n):
    """第 repeat_n 次请求（0=首次，>0=热身后）"""
    schema = SCHEMAS[schema_name]
    t0 = time.time()
    r = client.chat.completions.create(
        model=MODEL,
        messages=[{"role":"user","content":PROMPT}],
        max_tokens=256,
        response_format={"type":"json_schema",
                         "json_schema":{"name":schema_name,"schema":schema}}
    )
    elapsed = time.time() - t0
    try:
        parsed = json.loads(r.choices[0].message.content)
        valid = True
    except Exception:
        valid = False
    return elapsed, valid

if __name__ == "__main__":
    # 注意：一次只能运行一个框架（另一个不启动）
    # 根据当前启动的框架选择 client
    import sys
    framework = sys.argv[1] if len(sys.argv) > 1 else "vllm"
    client = client_vllm if framework == "vllm" else client_sglang

    print(f"\n框架: {framework}")
    print(f"{'Schema':<10} {'第1次(ms)':<12} {'第2次(ms)':<12} {'第3次(ms)':<12} {'合法率'}")
    print("-"*60)

    for sname in ("simple", "medium", "complex"):
        times = []
        valids = []
        for i in range(5):  # 跑5次：第1次含编译开销，后续热缓存
            t, v = single_request(client, sname, i)
            times.append(t * 1000)
            valids.append(v)
        legal = sum(valids) / len(valids)
        print(f"{sname:<10} {times[0]:<12.0f} {times[1]:<12.0f} {statistics.median(times[2:]):<12.0f} {legal:.0%}")
```

运行（先跑 vLLM，再换 SGLang）：
```bash
# 启动 vLLM，跑
NO_PROXY="*" python3 11_structured_output_deep.py vllm

# 切换 SGLang，跑
NO_PROXY="*" python3 11_structured_output_deep.py sglang
```

## 步骤 2：Compressed FSM 的优势验证（30 min）

构造一个"确定性字符极多"的 schema（强 pattern 约束），这是 SGLang 理论上更快的场景：

```python
# 加入下面这个 schema 到实验
SCHEMAS["high_det"] = {   # 高确定性：严格 pattern，enum 多
    "type":"object",
    "properties":{
        "sensor_id": {"type":"string", "pattern":"^SEN-[0-9]{6}-[A-Z]{3}$"},
        "status":    {"type":"string", "enum":["正常","警告","严重","停机"]},
        "value_hz":  {"type":"number", "minimum":0, "maximum":10000},
        "timestamp": {"type":"string", "pattern":"^[0-9]{4}-[0-9]{2}-[0-9]{2}T[0-9]{2}:[0-9]{2}:[0-9]{2}Z$"}
    },
    "required":["sensor_id","status","value_hz","timestamp"]
}
```

`pattern` 约束让大量字符位置是"确定的"（只有 0-9 或 A-Z），SGLang 的 jump_forward 应在这里体现优势（第 2+ 次请求的速度差异）。

**关注点**：
- 首次请求：两者都慢（FSM 编译），差异来自编译算法
- 热身后：通常差距 <10%（对 1.5B 小模型来说，LM forward 时间远大于 mask 计算时间）
- high_det schema：若 SGLang 有 jump_forward，热身后应更快；若看不出差异，说明 1.5B 模型的 decode 太快，FSM 开销被淹没了

## 步骤 4：结论笔记（10 min）

结构化输出的"速度损耗几乎为零"的正确理解：

- **正确**：对于热身后的请求，mask 计算在 CPU 上完成，与 GPU decode 并行；对于 1.5B 这种小模型，mask 计算时间（<1ms）远小于一次前向（几十 ms）
- **错误**：说"没有任何开销"——首次请求有编译开销（可能 0.5-5s），复杂 schema + 大 vocabulary 的 mask 预计算也需要时间
- **实际影响**：生产环境里 schema 固定，编译开销只发生一次，之后可忽略
