```python
# explore_fsm.py — 探索 FSM 工作原理
import outlines
from outlines.fsm.json_schema import build_regex_from_schema
from outlines.fsm.guide import RegexGuide
import json

SIMPLE_SCHEMA = {
    "type": "object",
    "properties": {
        "defect_type": {"type": "string"},
        "severity": {"type": "integer", "minimum": 1, "maximum": 5}
    },
    "required": ["defect_type", "severity"]
}

if __name__ == "__main__":
    # Step1: schema → 正则表达式
    regex = build_regex_from_schema(json.dumps(SIMPLE_SCHEMA))
    print(f"正则表达式（前200字符）:\n{regex[:200]}\n")
    print(f"正则总长度: {len(regex)} 字符")

    # Step2: 正则 → FSM（状态数）
    import re
    import interegular
    pattern = interegular.parse_pattern(regex)
    fsm = pattern.to_fsm()
    print(f"\nFSM 状态数: {len(fsm.states)}")
    print(f"初始状态: {fsm.initial}")
    print(f"接受状态数: {len(fsm.finals)}")

    # Step3: 模拟"在某状态下哪些字符合法"
    # 从初始状态出发，第一个字符只能是 '{'
    init = fsm.initial
    valid_first = [chr(c) for c in range(128)
                   if fsm.transition(init, c) is not None]
    print(f"\n初始状态下合法的首字符: {valid_first[:10]}...")
```

**核心发现要写进笔记**：
- schema 越复杂，正则越长，FSM 状态数越多（SIMPLE 可能 ~50 状态，COMPLEX 可能 ~500+）
- 状态数决定了 mask 计算开销：每步 decode 要用当前状态查合法 token 集合
- 这就是为什么 **首次请求慢**（编译正则→构建 FSM→预计算每个状态的 token mask），后续请求复用缓存的 mask 很快

## 步骤 4：SGLang Compressed FSM（30 min）

读 SGLang 的对应实现，重点找 "compressed" 或 "jump" 相关逻辑：

```bash
grep -rn "compressed\|jump_forward\|deterministic" $SGLANG --include="*.py" | head -10
```

**Compressed FSM 的核心优化**：对于"确定性"片段（只有一种合法输出，如 `{"name":"`），FSM 不用逐 token 走，可以**一步跳过整个片段**，直接到下一个"自由选择"状态。

类比：生成 `{"name":"` 这 9 个字符时，普通 FSM 要走 9 步，每步都 mask；Compressed FSM 识别出这段是确定的，一步跳过，节省 8 次前向传播。
