# 05-12 详细计划：社区讨论补充与交叉验证

## 步骤 1：定向搜索（60 min）

不要漫无目的地看，带着自己的数据和结论去找验证/反例。搜索四个方向：

**① GitHub Issues/PR（最有干货）**

```
vLLM GitHub: https://github.com/vllm-project/vllm/issues
搜索词: "prefix caching" / "block hash" / "radix" / "apc performance"

SGLang GitHub: https://github.com/sgl-project/sglang/issues  
搜索词: "radix cache" / "evict" / "cache hit rate"
```

重点找：有具体数字的 issue、PR 里关于设计取舍的讨论（比"为什么不用 radix 树"之类的决策讨论最有价值）

**② 官方 blog**

- vLLM: https://blog.vllm.ai
- SGLang: https://lmsys.org/blog （搜 "v0.4" / "radix" / "prefix")

摘录有具体 benchmark 数字的段落，记下测试条件（模型/硬件/负载）。

**③ 争议点：关于 PagedAttention 的批评**

Flashinfer 博客曾讨论 PagedAttention 的 kernel 效率问题，搜：
```
site:flashinfer.ai paged attention
```
核心争议：PagedAttention 的间接寻址让 GPU 访存不连续，kernel 效率低于连续布局——这和 04-27 的"误区纠偏"（PagedAttention 让显存更省，不是让计算更快）直接对应。

**④ 对比评测**

搜 "vLLM vs SGLang benchmark 2024" / "2025"，注意：
- 记录每篇的测试时间、模型规模、硬件（A100/H100 的结论未必适用你的 3060+1.5B）
- 找和你 05-10 数据趋势一致的——用来支持你的结论
- 找和你不一致的——标注可能原因

| 来源 | 结论 | 与我的数据 | 差异可能原因 |

| vLLM 官方 blog | APC 在 xxx% 复用率下提升 xx% | 一致/差异 | |
| SGLang issue #xxxx | radix 树在 xxx 场景下 xx% 提升 | | |

重点标注：**你在 3060+1.5B 上得到的结论，哪些能外推到大模型/多卡场景，哪些不能。** 这是分享时被质疑时最好的免责声明，也是诚实做研究的体现。
