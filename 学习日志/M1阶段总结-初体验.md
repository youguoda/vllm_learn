# M1 阶段总结 —— vLLM / SGLang 初体验

> M1 交付。两周（Week 1-2）从零跑通 vLLM 和 SGLang，每个核心机制都用**自己卡上跑出来的数据**验证。本文三段式：vLLM 三大支柱 → SGLang 两大特性 → 初步对比结论。
>
> 环境：RTX 3060 12GB (WSL2)，Qwen2.5-1.5B，vLLM 0.22.1 / SGLang 0.5.13。**所有数字均为本机实测，可复现。**

---

## 第一段：vLLM 三大支柱

vLLM 的高吞吐建立在三个互相配合的机制上：**PagedAttention 管显存、Continuous Batching 管调度、Prefix Caching 管复用**。

| 支柱                           | 机制一句话                                                              | 我的证据                                                                                                  |
| ------------------------------ | ----------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------- |
| **PagedAttention**       | KV 切成 16-token 块 + Block Table 间接寻址，碎片浪费从 98.8% 压到 21.9% | 块核账：14268 块 × 16 = 228,288 token，并发 55.73x（[[vLLM-PagedAttention实测-06-13]]）                  |
| **Continuous Batching**  | 逐 step 调度，完成的请求立即移除、新请求随到随插                        | 并发 1→5 吞吐涨**3 倍**（28.9→88.1 RPS），峰值 90.7 RPS（[[vLLM-Continuous-Batching实验-06-11]]） |
| **Prefix Caching (APC)** | 块哈希复用共享前缀的 KV，跳过重复 prefill                               | WARM TTFT**降 4.6 倍**（152.6→33.3ms），命中率 98.7%（[[vLLM-PrefixCaching实测-06-13]]）           |

### 1.1 PagedAttention：显存→块→并发这条链

亲手从 `/metrics` 抓到 `num_gpu_blocks=14268`、`block_size=16`，验算闭环：

```
可用 KV 显存 6.1GiB → 14268 块 → ×16 = 228,288 token → ÷4096 = 55.73x 并发
                     (实测)          (=日志值✅)          (=日志值✅)
```

**碎片对比**（10 个 50-token 请求）：传统预留 max_len 浪费 **98.8%**，PagedAttention 按需分块只浪费 **21.9%**。同样显存，传统放 55 个请求，PagedAttention 放 3567 个——**65 倍**。

> 关键认知：vLLM 快不是因为 kernel 更快，而是**同样显存能同时服务多一两个数量级的请求**。

### 1.2 Continuous Batching：吞吐 vs 延迟的权衡

|         并发 |                    RPS | Mean TTFT |
| -----------: | ---------------------: | --------: |
|            1 |                   28.9 |      61ms |
|            5 |                   88.1 |      38ms |
| **10** | **90.7（峰值）** |      44ms |
|           20 |                   73.9 |      71ms |

底层原理：decode 是**显存带宽瓶颈**——每生成 1 token 都要把整个模型权重读一遍，这个开销固定。batch 越大，单 token 的"权重搬运成本"摊得越薄 → 吞吐越高。最佳并发区间 5-10，超过后 GPU 算力饱和。

### 1.3 Prefix Caching：长前缀场景的利器

开/关 APC 对照（1000-token 共享前缀）：

|         |        WARM TTFT | 命中率 |
| ------- | ---------------: | -----: |
| APC ON  | **33.3ms** |  98.7% |
| APC OFF |          152.6ms |     0% |

且 **decode 时间几乎不变**（249 vs 220ms）——APC 只省 prefill(TTFT)，不省 decode。破坏实验：前缀加一个随机字符 → 命中率归零（链式哈希，差一 token 全失效）。

## 第二段：SGLang 两大特性

SGLang 的差异化在两点：**RadixAttention 把缓存粒度做到 token 级、Structured Output 用 FSM 保证 100% 合法输出**。

| 特性                        | 机制一句话                                           | 我的证据                                                                                                       |
| --------------------------- | ---------------------------------------------------- | -------------------------------------------------------------------------------------------------------------- |
| **RadixAttention**    | radix 树（page_size=1）索引 KV，任意长度前缀都能命中 | 多轮对话轮 2 起 TTFT 降**73%**；半块前缀命中 94.9% vs vLLM 82.1%（[[RadixAttention多轮对话实验-06-13]]） |
| **Structured Output** | schema 编译成 FSM，在 logits 层 mask 非法 token      | 合法率**0% → 100%**，速度损耗 ~0（[[SGLang结构化输出实验-06-13]]）                                      |

### 2.1 RadixAttention：缓存粒度才是本质优势

**多轮对话**（同 system prompt 连续追问）：前缀越长降幅越大。

| 前缀 token | 轮1 TTFT | 轮2-5 均值 |          降幅 |
| ---------: | -------: | ---------: | ------------: |
|        352 |     59ms |       28ms |           52% |
|        726 |     73ms |       29ms |           60% |
|       1452 |    110ms |       30ms | **73%** |

日志铁证：轮 2 起每轮**只 prefill 16-19 个新 token**，历史全在树上（cached-token 1570→1919）。

**半块对齐前缀对照（最关键的一张表）**：用完全相同的请求，对比两框架命中：

| 框架             | 机制                   |              命中 |          命中率 |
| ---------------- | ---------------------- | ----------------: | --------------: |
| vLLM             | 块哈希 block_size=16   |            96/117 |           82.1% |
| **SGLang** | RadixCache page_size=1 | **111/117** | **94.9%** |

> SGLang 多命中 **15 token**，正好是 vLLM 因 16-token 块边界对齐损失的部分。**这就是缓存粒度差异的直接证据**——共享前缀不在块边界上时，vLLM 尾部一整块白算，SGLang 逐 token 精确复用。

### 2.2 Structured Output：约束解码

| 方式        |                                                合法率 |
| ----------- | ----------------------------------------------------: |
| 无约束      | **0%**（1.5B 模型直接复述 prompt，没输出 JSON） |
| schema 约束 |                                        **100%** |

**对抗实验**：要求"用诗歌形式回答"，带 schema 后**仍输出合法 JSON**。原理：schema 编译成 FSM，每步解码前算出"合法的下一个 token 集合"，把非法 token 的 logits 置 **-∞**。合法性在采样前物理保证，与模型意愿无关——**想写诗也写不出来**。FSM 编译开销 ~0.28s（首次），之后缓存复用。

## 第三段：初步对比结论

### 3.1 无共享前缀的通用负载：两者接近

40 个不同主题（无前缀复用）压测，配置严格对齐（ctx=4096, mem=0.85）：

![compare](../assets/compare_v1.png)

| 并发 | vLLM RPS | SGLang RPS |
| ---: | -------: | ---------: |
|    1 |     0.72 |       0.89 |
|   10 |     6.63 |       6.96 |
|   20 |    10.73 |      10.05 |

**结论：基本持平**（RPS 差异在单次测量噪声范围内）。符合预期——这种负载下 PagedAttention/RadixAttention 都用不上。

### 3.2 已观察到的真实差异

1. **缓存粒度**：半块前缀实验，SGLang 命中 94.9% vs vLLM 82.1%（token 级 vs 块级）。
2. **结构化输出易用性**：SGLang `response_format` 开箱即用，xgrammar 后端，合法率 100%。

### 3.3 待验证假设

> **前缀复用率越高，SGLang 优势越大** —— 多轮对话实验已初步证实（轮2起 TTFT 降 73%），但需要在受控的"复用率梯度"实验里量化。→ Week 3 的 05-10 实验。

### 3.4 一句话选型建议（当前认知）

- **纯生成、无共享前缀**（如批量翻译、独立摘要）：两者接近，看生态/熟悉度。
- **高前缀复用**（多轮对话、RAG、固定 system prompt、few-shot）：**SGLang 缓存粒度优势明显**。
- **需要结构化输出**（Agent、工具调用、数据抽取）：两者都支持约束解码，SGLang 体验顺手。

---

## 诚实的局限声明

- 对比跑分是**单次**测量，差异 <10% 记为持平；正式结论需各跑 3 次取中位（Week 6）。
- 吞吐 tok/s 两框架**不可直接比**（生成长度不同），RPS 更公平。
- 并发 20 时 SGLang TTFT 偏高（135ms vs 58ms），疑似 `cuda_graph_max_bs=8` 限制，**待复测**，不下草率结论。
- 所有实验都是 1.5B 小模型 + 单卡，大模型/多卡行为可能不同。

## M1 收获总览

```
两周从零到跑通，9 篇实验笔记，全部自测数据：

vLLM:    PagedAttention(碎片 98.8%→21.9%) + Continuous Batching(并发1→5 涨3倍)
         + Prefix Caching(TTFT 降4.6倍)
SGLang:  RadixAttention(半块命中 94.9% vs 82.1%) + Structured Output(0%→100%)
对比:    无前缀持平, 高复用 SGLang 占优(待量化)

核心认知: 推理框架的快, 不在 kernel, 在【显存管理】和【缓存粒度】。
```

对外讲清楚 vLLM/SGLang，只需要"两张图一张表"。讲完检查听众能否复述三个问题。

```
                [■■■■·] [■■■■·] ← 块可散落各处, 只浪费最后半块

```

### 图 2：SGLang 缓存粒度细在哪（RadixAttention）

```

SGLang radix 树 (page_size=1): 任意 token 位置可 split
        root─"逐token精确匹配"→  → 命中 111/117 = 94.9%
                                    多救回 15 token (= 一个块的损失)
```

### 表：什么场景选谁

| 场景                                | 选择             | 依据                                           |
| ----------------------------------- | ---------------- | ---------------------------------------------- |
| 纯生成、无共享前缀                  | 都行             | 实测 RPS 持平                                  |
| 多轮对话 / RAG / 固定 system prompt | **SGLang** | 缓存粒度细，前缀越长省越多（轮2起 TTFT 降73%） |

### 试讲自检三问

听众听完能否复述：

1. **vLLM 怎么省显存？** → KV 分块 + Block Table 间接寻址，把连续预留的浪费变成小碎片。
2. **SGLang 缓存粒度细在哪？** → page_size=1 的 radix 树，任意 token 位置可 split，不受 16-token 块边界限制。
3. **什么场景该选谁？** → 高前缀复用选 SGLang，纯生成两者持平。

> 卡壳的地方 = M1 没消化透的地方，当场补。
