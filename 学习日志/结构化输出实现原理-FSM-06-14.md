# 结构化输出实现原理（FSM 深入）

> 衔接 05-03（约束解码实验：0%→100% 合法率）。今天深入"正则→FSM→token mask"这条链路的代码实现，并理解 SGLang 的 Compressed FSM 优化。

---

## 步骤 1：正则 → FSM → 合法字符（本地跑通）

用 `interegular`（vLLM/outlines 的依赖）把一个简化 JSON 正则编译成 FSM。[explore_fsm.py](../explore_fsm.py) 实测：

```
正则: \{"defect_type":"[a-z ]+","severity":[1-5]\}
FSM 状态数: 34
初始状态合法首字符: ['{']   ← 只有左花括号合法!

逐字符走 FSM:
  状态 0 --'{'--> 1   (合法字符数: 1)
  状态 1 --'"'--> 2   (合法字符数: 1)
  状态 2 --'d'--> 3   (合法字符数: 1)   ← "defect_type" 每步只有 1 个合法字符!
  状态 3 --'e'--> 4   (合法字符数: 1)
  ...
```

### 核心发现

1. **schema → 正则 → FSM**：schema 越复杂，正则越长，FSM 状态越多（简化版 34 状态，真实 COMPLEX schema 可达数百状态）。
2. **每个状态都知道"下一步哪些字符合法"**：初始状态只允许 `{`，`"defect_type"` 这段**每步只有 1 个合法字符**。
3. **这解释了首次请求慢**（05-03 实测 ~0.28s）：要编译正则→构建 FSM→预计算每个状态的 token mask，后续请求复用缓存。

## 步骤 2：vLLM 如何把 mask 注入 logits

源码 `vllm/v1/structured_output/`，4 个后端：`backend_xgrammar.py`（默认）/`backend_outlines.py`/`backend_guidance.py`/`backend_lm_format_enforcer.py`。

### 关键机制：bitmask（不是逐 token 比较）

```python
# backend_guidance.py:205
def fill_bitmask(self, bitmask: torch.Tensor, idx: int) -> None:
    llguidance_torch.fill_next_token_bitmask(self.ll_matcher, bitmask, idx)
```

- **bitmask**：词表大小的位向量，每个 token 1 位——合法=1，非法=0。
- vLLM 用 `fill_bitmask` 让 FSM 后端填这个 bitmask（当前状态下哪些 token 合法）。
- 应用：把 bitmask=0 的 token 的 logit 设为 **-inf**（采样前），模型只能从合法 token 里挑。
- 性能优化（`__init__.py:62`）：batch>128 时**并行填 bitmask**（`fill_bitmask_parallel_threshold`）。

### FSM 状态推进

```python
# backend_xgrammar.py:148
def accept_tokens(self, request_id, tokens) -> bool:
    """Accepts tokens and advances the FSM."""
```

> **完整循环**：每步 decode →（1）`fill_bitmask` 算当前状态的合法 token →（2）logits[非法]=-inf →（3）采样 →（4）`accept_tokens` 推进 FSM 状态。这就是 05-03 笔记说的"FSM 门禁"的代码实现。

## 步骤 3：SGLang Compressed FSM（跳跃式解码）

源码 `sglang/srt/constrained/outlines_backend.py`，关键是 `jump_forward_map`：

```python
# outlines_backend.py:84
jump_forward_bytes = self.jump_forward_map.jump_forward_byte(self.state)
```

### 核心优化：确定性段一步跳过

回看步骤 1 的发现——`"defect_type"` 这段**每步只有 1 个合法字符**。普通 FSM 要逐字符走 12 步，每步都跑一次完整 forward pass（很贵）；**Compressed FSM 识别出这是"确定性段"，一步跳过整段**，直接到下一个"自由选择"状态（如 `severity` 的值 1-5）。

```
普通 FSM 生成 {"defect_type":":
  走 14 步, 每步 forward pass + mask  ← 14 次模型前向

Compressed FSM:
  识别 {"defect_type":" 是确定段(每步唯一)
  → 一步 jump_forward 直接填进去
  → 只在真正"自由选择"处(defect_type 的值、severity)跑 forward
  → 省掉 ~13 次前向
```

> 这正是 05-03 笔记里画的"填表/直路踩油门"——固定格式部分直接写上，模型只在"空"上动脑。SGLang 论文实测 JSON decoding 吞吐提升约 **1.6×**。

---

## 步骤 4：vLLM vs SGLang 对比

| 对比点 | vLLM (xgrammar/outlines) | SGLang (Compressed FSM) |
|---|---|---|
| 每步处理 | 查 bitmask → logit 置-inf → 推进状态 | 同，但**确定性段一步跳过** |
| 首次耗时 | 编译 FSM + 预计算 mask | 同，还要分析跳跃点 |
| 重复 schema | 缓存 mask | 缓存 + 跳跃图 |
| 优势场景 | 通用 | schema 确定性字符多（严格 JSON key/标点） |
| 吞吐增益 | 基线 | JSON ~1.6× |

> **本质差异**：vLLM 逐 token 走 FSM（每个 token 一次 mask + forward）；SGLang 对"答案已注定"的确定段一次性跳过，省掉那些"只有一条路还要跑前向"的浪费。schema 里固定结构越多（JSON 的 key、引号、冒号、逗号），SGLang 省得越多。
