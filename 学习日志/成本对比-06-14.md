# 成本维度分析

> 用实测吞吐换算"每百万 token 成本"（[cost_analysis.py](../cost_analysis.py)）。成本是选型的现实维度。

---

## 每百万 token 成本（无共享前缀，满载吞吐）

实测最高吞吐：vLLM 1443 tok/s（5.20 M token/小时），SGLang 1342 tok/s（4.83 M/h）。

| GPU (RMB/小时) | vLLM | SGLang |
|---|---:|---:|
| RTX 3060级 (¥1.5/h) | ¥0.289/M | ¥0.310/M |
| RTX 4090级 (¥3.5/h) | ¥0.674/M | ¥0.724/M |
| A10 (¥5.0/h) | ¥0.962/M | ¥1.035/M |
| A100 80G (¥18.0/h) | ¥3.465/M | ¥3.725/M |

---

## 核心发现

### 1. 无共享前缀：成本差异 = 吞吐差异（8%）

vLLM 单位成本略低（吞吐高 8%），但同量级。**无复用场景两框架成本基本一致**。

### 2. 高复用场景：SGLang 成本可降一半

复用率 100% 时 SGLang 吞吐 2.1x（[[前缀复用率实验-06-14]]）→ **单位 token 成本约为 vLLM 的一半**。

> 对**统一 system prompt 的客服系统**（每个请求都带相同的几百 token 角色设定）：SGLang 把那段 KV 算一次全局共享，吞吐翻倍 → 同样的 GPU 处理两倍请求 → **成本减半**。这是 SGLang 在高复用生产场景的最大商业价值。

### 3. 量化进一步降本

- AWQ INT4 省 64% 显存 → 同卡能跑更大模型或更多 KV → 间接降本。
- FP8 KV cache 并发翻倍 → 单卡吞吐潜力翻倍 → 降本。

---

## 成本选型建议

| 场景 | 成本最优选择 |
|---|---|
| 通用/低复用 | vLLM（成本略低 + 生态成熟） |
| 高复用（客服/RAG） | **SGLang（成本可减半）** |
| 显存紧张 | 任意 + AWQ INT4 量化 |
| 长上下文/高并发 | 任意 + FP8 KV cache |

---

## 局限

- 成本数字基于 **1.5B 小模型单卡满载** + 假设的云 GPU 价格。
- 大模型吞吐更低 → 单 token 成本更高；具体价格随云厂商/时段波动。
- 仅算 **GPU 卡时**，未含运维/网络/存储等隐性成本。
