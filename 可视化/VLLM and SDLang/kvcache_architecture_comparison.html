维度vLLM（PagedAttention + APC）SGLang（RadixAttention）
缓存粒度 16-token Block，前缀必须对齐到块边界才能命中 单 token（page_size=1），任意长度公共前缀全部命中
索引结构 Block Table（逻辑块→物理块哈希表）+ Chain Hash Radix Tree（压缩前缀树，CPU维护，O(L)查找）
命中方式 Hash 精确匹配：block内容的哈希值相同才命中 前缀匹配：match_prefix 沿树走，走多远复用多长
驱逐策略 LRU（块级），由调度器统一管理 LRU（叶优先），lock_ref 保护运行中节点，父节点随子节点自然共享
显存布局 非连续物理块，Block Table 统一映射；支持 Copy-on-Write 共享 非连续 paged 布局，每 page = 1 token；树节点存 CPU，KV 存 GPU
多请求共享 相同 Block Hash → 物理块共享（需 APC 开启，块对齐） 自动：所有请求共用同一棵树，公共前缀天然共享节点
调度优化 Block-aware 调度；WAITING/RUNNING/PREEMPTED 三状态 Longest-Matched-Prefix-First（深度优先近似），理论最优命中率
最大优势场景 通用批量推理、长输出、无共享前缀；内存利用率高 多轮对话、RAG、Few-shot、同 system prompt 高并发；高复用率
典型命中率 APC 开启时：system prompt 场景 60-80% Chatbot Arena 实测：Vicuna-33B 74.1%，LLaVA-34B 52.4%
开销 Block Table 维护 O(1)；Chain Hash 计算在 block 填满时 树操作 O(L)，L=序列长度；实测额外开销 <0.3%
一句话总结：两者都解决了"KV Cache 如何高效复用"，但切入角度不同。vLLM 以块为单位做显存管理（类比 OS 虚拟内存），重点在显存利用率；SGLang 以 token 为粒度做前缀索引（类比文件系统前缀树），重点在最大化命中覆盖。两者差异在高复用负载下才会显现。
